# FoxChat 记忆架构优化实现计划（第二版）

> 基于 `docs/memory_architecture_optimization_plan.md` 与 `docs/memory_flow_recommended.md` 的讨论结果整理  
> 目标：在尊重当前代码现状的前提下，给出更严谨、更贴近最终设计的实施计划  
> 说明：本计划不写实现代码，但会明确对象、流程、阶段、风险与验证口径  
> 第一版优先保证语义正确性、跨日跟进能力与时间相关 case 的可解释性；Token 优化放到能力跑通之后，不作为阶段 0~3 的前置阻塞条件

## 一、当前架构分析总结

### 1.1 当前现状（按最新代码核对）

```text
┌─────────────────────────────────────────────────────────────────────┐
│                      当前记忆注入流程（最新现状）                     │
└─────────────────────────────────────────────────────────────────────┘

    Redis / Chroma 存储                     注入方式 / 当前状态
    ─────────────────────────────────────────────────────────────────
    soul / core_anchor ─────────────────────► 常驻注入 ✓
    character_card ─────────────────────────► 常驻注入 ✓
    user_profile ───────────────────────────► 常驻注入 ✓

    emotion_state ──────────────────────────► 已注入 ✓
    memory_bank (JSON数组) ─────────────────► 只注入最近5条 ✓

    Chroma CHAT (对话总结) ─────────────────► 已检索并注入 ✓
    recent_msg (原始消息) ──────────────────► history_msg ✓

    当前仍存在的问题：
    • memory_bank 仍是 event / state 混合结构
    • current_state 目前只有 emotion，维度不足
    • Chroma 检索已接入，但仍是最简单的 similarity_search
    • 长期禁忌 / 高优先级边界尚未提升为独立常驻层
```

### 1.2 与最初草案相比，需要修正的事实

当前代码已经发生以下变化：

1. `memory_bank` 不再全量注入，而是只取最近 5 条作为保底注入
2. `emotion_state` 已接入主 Prompt
3. `relevant_memories` 已经通过 `_search_relevant_memories()` 接入主 Prompt
4. `async_summary_msg()` 中 `_summary_and_upload()` 已重新启用

这意味着当前系统已经进入：

> **“保底 Memory Bank + 情绪状态 + 简单 Chroma 检索” 的过渡阶段**

因此，后续计划不应再按“完全没有检索 / 完全没有状态注入”的旧假设展开。

### 1.3 核心问题重新定义

当前问题已经不是“有没有开始做状态和检索”，而是：

1. **状态层不完整**：目前只有情绪，没有关系态势、当前焦点、未完成事项、互动模式
2. **历史层粒度不清**：memory_bank 中 event/state 仍混合，Chroma 中是对话总结，不是结构化事件
3. **检索层过于朴素**：目前只有简单向量检索，没有混合检索、去重、类型过滤、rerank
4. **层级职责未彻底拆开**：长期边界、当前状态、历史事件仍有交叉
5. **缺少最小时间语义能力**：系统能记住“说过什么”，但还不能可靠处理“什么时候该重新想起来”，导致“明天考试 / 下周出结果 / 明天继续聊”这类跨日跟进场景会系统性失败

## 二、推荐的目标架构定义

本计划采用讨论后收敛的“常驻层 + 当前状态层 + 历史检索层 + 最近窗口 + 时间节点运行时机制”，而不是旧版的粗粒度“三层”表述。

```text
A. 常驻稳定层
   A1. 静态锚点
      - soul
      - core_anchor
      - character_card

   A2. 稳定画像 / 关键边界
      - user_profile
      - 长期偏好
      - 永不衰减禁忌
      - 高优先级边界

B. 当前工作状态层
   - emotion
   - relation_state
   - current_focus
   - unfinished_items
   - interaction_mode

T. 时间节点层（运行时激活，不常驻注入）
   - future commitments
   - due follow-ups
   - time-bound reminders

C. 历史事件检索层
   - 过去重要事件
   - 承诺 / 冲突 / 共同经历
   - 旧话题锚点

D. 最近对话窗口
   - 最近 4~6 轮原始消息
```

### 2.1 层级职责说明

#### A1. 静态锚点
职责：固定角色身份、风格与底线。  
特点：基本不变、每轮常驻、权重最高。

#### A2. 稳定画像 / 关键边界
职责：表达长期稳定的用户画像、偏好与禁忌。  
特点：会随总结缓慢更新，但仍应每轮常驻，不应依赖检索召回。

#### B. 当前工作状态层
职责：表达“现在是什么局面，这一轮应如何继续回应”。  
特点：轻量、常驻、更新频率高于 A2。

#### T. 时间节点层
职责：保存“未来某个时间点应该重新浮现”的事项，并在到期时激活。  
特点：默认挂起、不常驻注入、按当前时间与节点条件在运行时触发；它不是完整时间线系统，而是最小的跨日跟进能力。

#### C. 历史事件检索层
职责：按需回忆与当前输入相关的过去事件。  
特点：检索式、限量注入、不应承担当前状态职责。

#### D. 最近对话窗口
职责：保证近程上下文、语气、指代与连贯性。  
特点：工作窗口，不属于长期记忆。 

### 2.2 时间节点层与其他层的关系

时间节点层不是 A2 / B / C / D 的替代层，而是一个横向运行时机制：

- 它**不负责长期画像**，因此不进入 A2。
- 它**不直接等同于当前状态**，而是在到期时把相关事项激活到 B 层或辅助触发 C 层。
- 它**不等同于历史事件检索**，因为它的触发依据首先是“时间到没到”，而不是“语义像不像”。
- 它**不常驻注入 Prompt**，只在节点到期且与当前轮有意义时产生可见影响。

可将其理解为：

```text
过去提到的未来事项
   ↓
生成 time node
   ↓
pending 挂起
   ↓
到期后 runtime 激活
   ↓
进入 current_state / 辅助历史检索触发
```

## 三、设计定义（四项核心工作）

### 3.0 术语约定（第一版统一口径）

为避免“轮”“消息”“总结批次”在不同阶段被不同理解，第一版统一采用以下定义：

- **1 条消息（message）**：指一条实际聊天消息，主体可能是 USER 或 AI。
- **1 轮（round）**：默认指 **1 次用户输入及其对应的 1 次 AI 回复**。若某次用户输入尚未产生回复，则该轮未闭合，不计入稳定统计窗口。
- **最近 4~6 轮**：指最近 4~6 个**已闭合 round**，不是最近 4~6 条单独消息。
- **`expire_rounds`**：指某状态在经历多少个**已闭合 round**后失效，而不是经过多少条消息后失效。
- **事件去重中的“6 轮 / 30 轮”**：同样指已闭合 round 数，而不是单边消息计数。
- **若某实现链路暂时只能拿到 message 级计数**，则需要先映射成 round 再参与状态过期、A2 稳定判断与事件窗口判断，避免同一文档内混用两套口径。
- **1 个 summary batch**：指一次 `_summary_and_upload()` 或等价总结流程实际处理的一段对话窗口产物，而不是抽象概念上的“某几轮”。
- **连续 2 个 summary batch**：指两次相邻、成功落库的总结产物都提取出同一结论；若中间某次总结失败、跳过或窗口为空，则不计为连续。
- **summary batch 的推荐窗口**：第一版建议按最近 **6~12 个已闭合 round** 或接近当前总结触发阈值的一段窗口执行；若实现上采用 token/长度阈值触发，也应在产物 metadata 中记录覆盖 round 范围。
- **time node**：指从对话中提取出的“未来需要在某个时间点重新浮现”的挂起事项，不是历史事件本体，也不是常驻当前状态。
- **`due_at`**：指 time node 的归一化时间锚点；若用户只表达“明天 / 下周 / 晚上”这类粗粒度时间，则应保留相应精度，而不是伪造精确时间。
- **`precision`**：指时间精度，第一版推荐至少支持 `day` 与 `datetime` 两档。
- **`trigger`**：指系统在新一轮开始时，根据当前时间判断某个 time node 是否应从 `pending` 进入 `active` 的过程；第一版不要求后台定时器主动触发。

这样定义的原因是：状态延续、事件续写与用户体验更接近“问答轮次”而不是“单条消息数”，也更适合与最近窗口、承诺跟进、时间到期激活和检索触发逻辑统一。

### 3.1 当前状态层最小字段集合

**设计原则：** 小而精，常驻注入，表达“当前有效状态”，不替代长期历史；时间相关的未来激活逻辑由 time node 负责，不把 B 层直接做成完整时间管理容器。

```json
{
  "current_state": {
    "emotion": {
      "value": "安抚中",
      "confidence": 0.9,
      "expire_rounds": 3,
      "update_reason": "用户表达明显失落，角色回复进入安慰语气"
    },
    "relation_state": {
      "value": "亲近",
      "confidence": 0.75,
      "expire_rounds": -1,
      "update_reason": "用户连续多轮分享私密情绪，关系信任增强"
    },
    "current_focus": {
      "value": "考试压力",
      "confidence": 0.85,
      "expire_rounds": 2,
      "update_reason": "最近两轮都围绕考试结果与焦虑展开"
    },
    "unfinished_items": [
      {
        "content": "答应明天继续聊考试结果",
        "confidence": 0.95,
        "expire_rounds": 6,
        "update_reason": "角色明确承诺下轮继续跟进",
        "status": "pending"
      }
    ],
    "interaction_mode": {
      "value": "陪伴 / 安慰",
      "confidence": 0.8,
      "expire_rounds": 3,
      "update_reason": "当前对话目标是情绪安抚，而非争论或计划确认"
    },
    "last_update": "2026-04-26T10:30:00",
    "update_source": "summary_or_runtime"
  }
}
```

#### 当前状态层字段优先级

| 优先级 | 字段 | 理由 |
|--------|------|------|
| P0 | emotion | 已存在，可直接升级为 current_state 的一部分 |
| P0 | relation_state | 决定当前互动关系气氛 |
| P1 | current_focus | 防止忘记当前正在围绕什么展开 |
| P1 | unfinished_items | 承诺 / 计划 / 等待回应不能忘 |
| P2 | interaction_mode | 提升回复方向一致性 |

#### 当前状态层设计要求

- 允许不同字段有不同过期策略
- `confidence` 参与状态覆盖判断
- `update_reason` 默认用于调试，不默认完整注入 Prompt
- 最终注入给模型的是**状态摘要**，不是原始 JSON

#### current_state 摘要输出契约（用于阶段 2.7）

- 输出形式固定为**结构化短文本块**，不是原始 JSON，也不是自由发挥的一句话总结。
- 输出顺序固定为：`emotion -> relation_state -> current_focus -> unfinished_items -> interaction_mode`。
- 只注入**未过期**且 `confidence >= 0.6` 的字段。
- `unfinished_items` 最多注入 2 条，并优先保留 `status=pending` 且与当前轮相关的事项。
- `update_reason`、`last_update`、`update_source` 默认不注入 Prompt，仅用于调试和覆盖判断。
- 总预算目标控制在 **80~150 tokens**；若超过预算，先裁剪 `interaction_mode`，再裁剪低相关 `unfinished_items`。
- 若没有任何有效字段，则返回空字符串，不注入“当前状态为空”之类的占位文案。

**推荐输出样式：**

```text
【当前状态】
- 情绪：安抚中
- 关系：亲近
- 焦点：考试压力
- 未完成：明天继续跟进考试结果
- 互动方式：陪伴 / 安慰
```

**字段压缩规则（第一版）：**
- `emotion` / `relation_state` / `current_focus` / `interaction_mode` 只保留 `value`
- `unfinished_items` 只保留 `content`
- 不注入 `confidence` 数值，避免增加噪音
- 若 `current_focus` 与当前用户输入高度重复，可省略该字段，减少重复注入

**Redis Key:** `chat:memory:{user_id}:{llm_id}:current_state`

#### 当前状态层更新数据来源（补充）

当前状态层不能等到阶段 3 的完整提取链路完成后才开始工作，否则阶段 2 会出现“有容器、无稳定写入来源”的断层。
因此阶段 2 需要先采用**双来源过渡方案**，阶段 3 再切换到正式路由：

1. **runtime 来源（阶段 2 即可接入）**
   - `emotion`：继续复用现有情绪分类结果
   - `current_focus`：根据当前用户输入 + 最近 2~4 轮消息做轻量规则提取
   - `unfinished_items`：根据 AI 回复中的显式承诺、约定、待跟进事项做规则提取；若 time node 已到期，也允许把对应事项激活到这里
   - `interaction_mode`：根据当前轮对话意图做轻量分类
   - `relation_state`：阶段 2 先仅允许保守更新，只在高置信度场景下改写
   - `time_node_activation`：在组装当前轮记忆前，先做一次到期检查，把已到期节点转成 B 层候选或 C 层触发信号

2. **summary 来源（阶段 3 正式接入）**
   - 使用 `current_state_extractor.md` 对一段对话窗口做结构化提取
   - 与 runtime 候选合并后，再走统一覆盖逻辑

3. **阶段边界要求**
   - 阶段 2 的目标不是把所有状态字段都提取得很完美，而是先让 `current_state` 有稳定写入、覆盖、过期与注入链路
   - 阶段 3 再把“状态候选提取质量”和“状态 / 事件分流质量”补齐

#### 阶段 2 推荐最小实现口径

- `emotion`：直接复用现有 `emotion_state` 更新链路
- `current_focus`：优先取当前输入的主话题；若连续两轮主题一致则延续旧值
- `unfinished_items`：只记录明确承诺、明确等待、明确下轮继续的事项；若 time node 到期，则允许把对应待跟进项提升为当前轮有效事项
- `interaction_mode`：第一版只保留有限枚举，例如“闲聊 / 安慰 / 陪伴 / 解释 / 计划确认 / 轻微冲突”
- `relation_state`：第一版只在“明显升温 / 明显缓和 / 明显疏离”时变更，避免抖动
- `time node`：阶段 2 只要求完成最小运行时接线，不要求此阶段就把所有自然语言时间表达都提取得很完美

#### 当前状态字段取值口径（第一版推荐枚举）

1. **`interaction_mode` 推荐枚举**
   - `闲聊`
   - `安慰`
   - `陪伴`
   - `解释`
   - `计划确认`
   - `轻微冲突`
   - `鼓励`
   - `倾听`

   说明：
   - 第一版建议优先从这组有限枚举中取值，避免自由文本导致覆盖与评估困难。
   - 若实际提取结果不适合上述任一项，优先回退到最接近的上位枚举，而不是临时新增值。

2. **`relation_state` 推荐枚举**
   - `疏离`
   - `中性`
   - `亲近`
   - `紧张`
   - `缓和中`

   说明：
   - 第一版不追求细粒度关系建模，只表达当前互动氛围。
   - 若单轮信号不足，不应频繁在这些值之间来回跳变。

3. **`emotion.value` 口径**
   - 若现有 `emotion_state` 已有固定标签，则优先复用原标签体系，不重新发明一套。
   - 若后续发现现有标签过细，可在注入摘要阶段映射到更稳定的展示值，但底层存储仍保持与现有链路兼容。

4. **`current_focus` 口径**
   - `current_focus` 不采用严格枚举，而采用**短语级主题表达**，建议控制在 4~12 个字内。
   - 第一版优先表达“当前这几轮在围绕什么”，而不是抽象成人设或长期画像。
   - 若当前输入没有稳定主题，可为空，不强制生成。

5. **`unfinished_items.status` 推荐枚举**
   - `pending`
   - `done`
   - `cancelled`

   说明：
   - 第一版只保留这三个值，避免引入 `paused`、`unknown` 等过多中间态。
   - 只要事项不再需要后续跟进，就应从 `pending` 切到 `done` 或 `cancelled`。

6. **实现建议**
   - `interaction_mode`、`relation_state`、`unfinished_items.status` 建议在 schema 层直接限制枚举。
   - `emotion.value` 与 `current_focus` 允许自由文本或既有标签，但在 Prompt 注入层应做裁剪和归一化。
   - 阶段 7 评估时，状态稳定性优先看这些字段是否在合理范围内稳定，而不是追求语义描述越来越细。

### 3.2 时间节点对象最小字段集合

**设计原则：** 只解决“什么时候该重新想起来”，不尝试在第一版引入完整时间线管理。

```json
{
  "time_node_id": "tn_20260426_001",
  "content": "跟进考试结果",
  "due_at": "2026-04-27",
  "precision": "day",
  "trigger_mode": "on_due",
  "status": "pending",
  "created_from": "user_future_event",
  "source_round": 42,
  "last_triggered_at": null,
  "trigger_count": 0,
  "created_at": "2026-04-26T10:30:00",
  "updated_at": "2026-04-26T10:30:00"
}
```

#### 时间节点字段说明

| 字段 | 作用 |
|------|------|
| `time_node_id` | 调试、追踪、状态流转 |
| `content` | 到期后应重新浮现的事项 |
| `due_at` | 归一化时间锚点 |
| `precision` | 时间精度，决定触发判断方式 |
| `trigger_mode` | 到期即触发或到期后可持续触发 |
| `status` | `pending / active / done / cancelled / expired` |
| `created_from` | 来源，如 `user_future_event` / `ai_commitment` |
| `source_round` | 回溯来源轮次 |
| `last_triggered_at` | 防止同一轮或短时间内重复激活 |
| `trigger_count` | 观察节点是否长期悬挂未解决 |

#### 时间节点状态机（第一版）

```text
pending -> active -> done
             │
             ├-> cancelled
             └-> expired
```

#### 时间节点设计要求

- time node 只处理**未来需要跟进 / 未来会到期 / 未来应重新提起**的信息。
- 纯过去背景信息、无后续价值的时间描述不建节点。
- 第一版不要求节点主动推送，只要求在**新一轮用户消息到来时**完成到期检查。
- `active` 不等于立刻删除；只有在事项完成、取消或失去意义后才终结。
- time node 不直接作为一个新的常驻 Prompt 块注入，而是转成：
  - B 层的可跟进 `unfinished_items` / 当前提示；或
  - C 层检索触发信号。

#### 时间节点创建范围（第一版）

**推荐创建的情况：**
- 用户明确提到未来事件：如“明天考试”“下周出成绩”
- 用户或 AI 明确约定后续继续：如“明天继续聊这个”“后天再看结果”
- 某事项天然带有到期后的跟进价值：如复诊、结果公布、面试反馈

**默认不创建的情况：**
- 纯过去经历：如“我昨天很难过”
- 只有时间背景、没有未来跟进价值：如“前年我搬家了”
- 只对本轮短暂有效的表达：如“今天先不聊这个”

#### 时间节点触发规则（第一版）

1. **触发时机**
   - 每次收到新的用户输入时，扫描 `status=pending` 的 time node。
   - 若节点已到期，则转为 `active` 或视为“本轮激活”。

2. **按精度判断是否到期**
   - `precision=day`：当前日期 `>= due_at` 即可视为到期。
   - `precision=datetime`：当前时间 `>= due_at` 才视为到期。

3. **触发后的去向**
   - 若节点本质上是承诺 / 待跟进事项，优先激活到 B 层 `unfinished_items`。
   - 若节点更像“到期后需要回忆的旧事件”，则可作为 C 层历史检索的强触发信号。
   - 必要时允许 B + C 联动，但不应生成重复的大段注入。

4. **触发后不立即删除**
   - 若本轮只是到期但尚未真正跟进，节点保留为 `active`。
   - 若事项已明确完成，转 `done`。
   - 若用户明确说不用再提，转 `cancelled`。
   - 若超过宽限期仍不再有跟进意义，转 `expired`。

5. **第一版复杂度约束**
   - 第一版不要求处理复杂依赖、循环提醒、多时区主动调度。
   - 第一版只要求支持“单点到期 + 新一轮开始时检查 + 有限状态流转”。

#### time node 最小规则表（用于第一版直接落地）

##### 1. 创建规则表

| 输入模式 | 是否创建节点 | 推荐 `created_from` | 说明 |
|----------|--------------|---------------------|------|
| 用户明确未来事件：`明天考试` / `下周出结果` | 是 | `user_future_event` | 未来时间 + 事项本体明确 |
| 用户明确要求后续跟进：`明天提醒我问结果` / `明天继续聊` | 是 | `user_future_followup` | 未来时间 + 跟进行为明确 |
| AI 明确承诺后续继续：`明天我再陪你看结果` | 是 | `ai_commitment` | 需要保证后续可召回 |
| 只有过去背景：`我昨天很难过` | 否 | - | 这是历史信息，不是未来节点 |
| 只有模糊长期时间：`以后再说` / `改天吧` | 第一版默认否 | - | 时间锚点过弱，先不做 |
| 只有当前轮局部拒绝：`今天不想聊这个` | 否 | - | 属于当前状态，不是 time node |

##### 2. 时间归一化规则表

| 原始表达 | 推荐 `due_at` | `precision` | 第一版处理原则 |
|----------|---------------|-------------|----------------|
| 明天 | 次日日期 | `day` | 不伪造具体小时 |
| 后天 | 后两日日期 | `day` | 只保留日期精度 |
| 下周 | 下一个自然周的锚点日期或周起始日期 | `day` | 需在实现中固定一种统一口径 |
| 今晚 8 点 | 具体 datetime | `datetime` | 保留明确时间 |
| 明天下午 | 次日日期 | `day` | 第一版可降级，不强做半天精度 |
| 出成绩那天 / 面试结束后 | 暂不归一化成功 | - | 第一版若无法稳定解析，宁可不建节点 |

**归一化总原则：**
- 第一版只要求稳定支持 `day` 与 `datetime`。
- 不能高置信归一化时，优先不建节点，而不是伪造时间。
- 同类表达必须在实现中固定统一口径，避免“同一句话今天解析成日期、明天解析成时间”。

##### 3. 到期后去向规则表

| 节点类型 | 到期后优先去向 | 是否辅助触发 C 层 | 说明 |
|----------|----------------|--------------------|------|
| 承诺继续 / 待跟进事项 | B 层 `unfinished_items` | 可选 | 如“明天继续聊考试结果” |
| 需要想起某个旧结果/旧背景 | C 层历史检索 | 是 | 如“下周出成绩”可能需要先回忆背景 |
| 同时带有跟进和背景依赖 | B + C | 是，但要去重 | 可激活待办，同时拉起背景事件 |
| 纯提醒但无聊天价值 | 第一版默认不处理 | 否 | 避免把系统做成泛提醒器 |

##### 4. 终止规则表

| 条件 | 目标状态 | 说明 |
|------|----------|------|
| 已明确跟进完成 | `done` | 如结果已聊完、事项已结束 |
| 用户明确取消 | `cancelled` | 如“不用了”“别再提醒了” |
| 超过宽限期且明显失去意义 | `expired` | 如过了很久仍无必要继续提 |
| 只是到期但本轮尚未解决 | 保持 `active` | 不应因为触发过一次就立刻删除 |

##### 5. 第一版推荐判定顺序

```text
收到用户新消息
  ↓
扫描 pending time nodes
  ↓
判断是否 due
  ↓
将到期节点标记为 active
  ↓
判断去向：优先进 B 还是辅助触发 C
  ↓
组装本轮 prompt
  ↓
回合结束后根据结果决定：保留 active / done / cancelled / expired
```

##### 6. 第一版实现红线

- 不把 time node 直接做成常驻 Prompt 大块。
- 不把所有带时间词的话都塞进节点。
- 不因节点触发一次就立刻删除。
- 不为了覆盖极少数模糊表达而引入复杂时间线系统。
- 不在阶段 2 就引入循环提醒、多时区计划器或后台主动调度。

### 3.3 历史事件对象最小字段集合

**设计原则：** 可检索、可判断新旧、可区分主体、可去重。

```json
{
  "event_id": "evt_20260426_001",
  "occurred_at": "2026-04-26T10:30:00",
  "last_seen_at": "2026-04-26T10:30:00",
  "actor": "USER",
  "type": "event",
  "event_type": "share_experience",
  "content": "用户分享自己最近要考试，感到压力大",
  "keywords": ["考试", "压力", "学习"],
  "importance": 0.8,
  "source_round": 42,
  "source_snippet": "我最近要考试了，压力特别大……",
  "activity_score": 0.9,
  "tags": ["学业", "压力表达"]
}
```

说明：
- 第一版推荐使用 `occurred_at` + `last_seen_at`，采用 ISO datetime，避免只有日期精度时无法支撑“2 小时内重复”这类判断。
- 若暂时只能拿到日期级信息，必须在实现中标记为低精度时间，并避免启用小时级重复窗口。
- `occurred_at` 表示首次发生时间，`last_seen_at` 用于续写合并后的最近一次出现时间。
- 若后续仍需保留旧字段名 `time` 兼容现有链路，也应在文档中明确其语义等价于 `occurred_at`，而不是同时混用两套定义。


#### 历史事件对象字段优先级

| 优先级 | 字段 | 用途 |
|--------|------|------|
| P0 | event_id | 追踪、调试、引用 |
| P0 | occurred_at / last_seen_at | 判断新旧、去重窗口、续写合并 |
| P0 | actor | 明确主体 |
| P0 | type / event_type | 区分大类与细类 |
| P0 | content | 事件本体 |
| P1 | keywords | 支持关键词召回 |
| P1 | importance | 排序优先级 |
| P1 | source_snippet | 保留少量原文，防止摘要失真 |
| P2 | source_round | 回溯调试 |
| P2 | activity_score | 后续支持事件活性衰减 |

#### 历史事件层设计要求

- `event_id` 只用于追踪，不单独承担去重职责
- 去重应基于：`actor + type + 时间邻近 + content 相似度 + keywords`
- 不同事件类型采用不同衰减策略
- 明确禁忌类事件不只存历史层，还应提升到 A2 层

**第一版存储策略：**
- 保留当前 Chroma CHAT 集合作为过渡方案
- 结构化事件写入时增加 metadata：`user_id, llm_id, event_id, actor, type, event_type, occurred_at, last_seen_at, importance`
- 后续若摘要检索与事件检索互相污染，则拆分为独立 collection

### 3.4 A2 稳定画像 / 长期禁忌的实现路径

A2 在目标架构中已经被提升为常驻层，因此不能只停留在“概念存在”，必须有独立的信息来源、存储位置、更新规则与注入表达。

#### A2 建议拆成两个子部分

1. **稳定画像（可缓慢更新）**
   - 现有 `user_profile`
   - 长期偏好
   - 稳定互动习惯

2. **关键边界（高优先级常驻）**
   - 长期禁忌
   - 明确拒绝的称呼 / 话题 / 互动方式
   - 永不衰减或极慢衰减的高优先级限制

#### A2 的推荐实现路径

**阶段 1：定义对象契约**
- 在 `docs/layer_definition.md` 和 `docs/memory_routing.md` 中明确：
  - 哪类信息属于稳定画像
  - 哪类信息属于长期禁忌 / 高优先级边界
  - 哪类负面事件应从历史层提升到 A2

**阶段 3：在总结流程中加入 A2 候选提取**
- 从对话窗口中额外提取：
  - 长期偏好候选
  - 长期禁忌候选
  - 高优先级边界候选
- 这些候选不进入 `memory_bank`，而是单独走 A2 路由判断

**阶段 3：实现 A2 路由与提升逻辑**
- 第一版不采用“模型自由判断是否长期稳定”，而是采用**规则优先的判定矩阵**。
- 若只是一次性情绪化表达、上下文性抱怨、短期偏好，则不进入 A2，留在 B 或 C 层。

#### A2 提升判定矩阵（第一版）

1. **立即进入 A2 的情况（无需重复出现）**
   - 用户明确表达高优先级边界，例如：
     - 不要使用某个称呼
     - 不要谈某类话题
     - 不要使用某种互动方式
     - 明确的隐私 / 身体 / 亲密边界
   - 这类信息的判断信号以**用户直接表达**为主，不要求多轮重复确认。

2. **延迟进入 A2 的情况（需要稳定性证据）**
   - 长期偏好、稳定互动习惯、长期画像特征，不因单次提取即进入 A2。
   - 第一版满足以下任一条件才提升：
     - 连续 **2 个 summary batch** 提取出同一结论；或
     - 最近 **30 轮** 内出现 **>= 3 次** 一致表达，且没有明显冲突证据。

3. **不进入 A2 的情况**
   - 一次性情绪发泄
   - 上下文性的短期抱怨
   - 仅对当前轮有效的拒绝或偏好
   - 随任务变化而变化的临时需求

4. **高风险踩雷的判定方式**
   - 第一版不依赖人工逐条审核，也不依赖模型自由发挥解释。
   - 采用固定规则类目优先识别：称呼禁忌、创伤 / 触发话题、隐私边界、身体 / 亲密边界、明确拒绝的互动方式。
   - 若命中上述类目，且来源是用户明确表达，则按高优先级边界处理。

5. **冲突与降级规则**
   - 若新候选与已有 A2 信息冲突，不直接覆盖，先标记为冲突候选并等待下一次 summary batch 或再次明确表达确认。
   - 对稳定偏好类 A2，允许后续因多次反向证据而降级回普通 profile；对高优先级边界类 A2，默认不自动降级。

6. **实现原则**
   - `user_explicit` 来源高于 `summary_inference`。
   - “长期稳定”是**重复证据规则**，不是抽象语义判断。
   - “高风险踩雷”是**命中固定规则类目**，不是人工临场主观评估。

**阶段 4~6：接入 Prompt 常驻层**
- 阶段 4 不需要改 Prompt 结构，但可以先保证相关 metadata 已可用于识别 A2 候选
- 阶段 6 在 Prompt Layout V2 中把 A2 作为独立常驻块显式注入

#### A2 推荐存储形态

可继续复用 `user_profile` 作为过渡容器，但建议逐步拆出“边界子结构”，例如：

```json
{
  "stable_profile": {
    "preferences": ["喜欢被温柔回应"],
    "interaction_habits": ["倾向先倾诉再接受建议"]
  },
  "hard_boundaries": [
    {
      "content": "不要叫用户宝宝",
      "confidence": 0.98,
      "source": "user_explicit",
      "priority": "high"
    }
  ]
}
```

#### A2 对象字段契约（第一版）

第一版建议将 A2 分为 `stable_profile` 与 `hard_boundaries` 两个子结构，并统一以下字段口径：

1. **`stable_profile` 推荐字段**
   - `preferences`: `List[str]`
   - `interaction_habits`: `List[str]`
   - `confidence`: 可选，表示该组稳定画像整体置信度
   - `source`: 推荐记录为 `summary_inference` / `user_explicit`
   - `last_updated_at`: ISO datetime

2. **`hard_boundaries` 单项推荐字段**
   - `content`: 边界内容文本
   - `category`: 推荐枚举为 `称呼禁忌` / `话题禁忌` / `互动方式禁忌` / `隐私边界` / `身体或亲密边界` / `其他`
   - `confidence`: 0~1
   - `source`: `user_explicit` / `summary_inference`
   - `priority`: 第一版固定为 `high` 或 `critical`
   - `status`: 推荐枚举为 `active` / `inactive`
   - `created_at`: ISO datetime
   - `last_updated_at`: ISO datetime
   - `evidence`: 可选，保留一小段来源摘要或 source_snippet 引用

3. **字段语义要求**
   - `hard_boundaries.status=active` 才参与常驻注入。
   - 高优先级边界默认**不使用 `expire_rounds` 自动衰减**；若后续要失效，应通过显式冲突确认或人工/规则降级处理。
   - `stable_profile` 可被缓慢更新；`hard_boundaries` 默认更保守，不应因单次弱证据被覆盖。

4. **`priority` 使用口径**
   - `critical`：明确且高风险踩雷，命中后必须优先约束回复。
   - `high`：明确边界或强偏好，默认常驻，但在极少数场景下可被更高层安全/系统约束覆盖。
   - 第一版不建议继续细分到 `medium` / `low`，避免 A2 失去“高优先级层”的意义。

5. **`status` 使用口径**
   - `active`：当前有效，参与常驻注入。
   - `inactive`：已被明确撤销、冲突证据确认、或人工下线，不再参与注入，但可保留作审计与调试。

6. **兼容与迁移建议**
   - 第一版可以继续复用 `user_profile` 作为存储容器，但写回时必须把 `stable_profile` 与 `hard_boundaries` 在结构上明确区分。
   - 若现有链路只支持扁平结构，至少在字段命名上区分出边界类信息，避免后续再次混入普通画像。

7. **注入建议**
   - Prompt 注入时优先输出 `hard_boundaries`，再输出 `stable_profile`。
   - `hard_boundaries` 默认只注入 `active` 且 `priority in {high, critical}` 的项。
   - `stable_profile` 注入时应做摘要压缩，不应把全部原始列表无裁剪展开。

第一版不要求立刻拆新 Redis key，但至少要在文档中明确：
- A2 不是普通 history event
- A2 不依赖检索召回
- A2 需要常驻注入

### 3.4 注入预算与评估标准

#### 第一版预算原则

- 第一版预算只作为**观测指标**，不作为阶段 0~3 的设计前置约束。
- 在 correctness 尚未跑通前，不为了省 token 提前压缩掉必要语义。
- time node 本身不作为常驻块注入，因此它新增的主要成本来自：
  - B 层被激活的待跟进事项
  - C 层被辅助触发的历史检索结果
- 阶段 6 以后再集中处理预算收口、重复裁剪与布局优化。

#### Token 预算建议（用于后续收口，不作为前置阻塞）

```text
┌─────────────────────────────────────────────────────────────────────┐
│                    推荐的动态注入预算（第一版）                      │
└─────────────────────────────────────────────────────────────────────┘

A1 静态锚点层：固定预算
A2 稳定画像 / 关键边界：固定预算
B 当前状态层：其中状态摘要本体建议 **80~150 tokens**；若包含少量冲突优先级说明与过渡期提示，整块建议控制在 **120~220 tokens**，第一版尽量不要超过 **250 tokens**
C 历史事件检索层：2~4 条事件，总计 ~300~600 tokens
D 最近对话窗口：最近 4~6 轮，尽量控制在 ~1200~1600 tokens

说明：上面的 B 层预算与 `current_state` 摘要输出契约保持一致；若阶段 6 将冲突规则从 B 层正文中拆到固定模板，则 B 层应进一步收敛到接近 **80~150 tokens**。
```

#### 评估指标

| 指标 | 当前值 | 第一版目标 | 测量方法 |
|------|--------|------------|----------|
| 历史事件召回正确率 | 未测量 | >80% | 人工标注样本 |
| 当前状态一致性 | 未测量 | >90% | 连续对话验证 |
| time node 激活正确率 | 未测量 | >80% | 构造时间样本 + 人工标注 |
| time node 误触发率 | 未测量 | 尽量低且可解释 | 构造时间样本 + 日志 |
| 每轮总 Token | 未严格基线化 | 可解释，不作为前置硬门槛 | API usage / 日志 |
| 响应时延 | 未测量 | 尽量不明显增加 | 对比测试 |

注意：当前值必须先做阶段 0 基线采样，不应直接凭估算写死。第一版先看 correctness，再看 token。

## 四、推荐 Prompt 注入顺序

本计划采用讨论后确认的顺序，而不是旧版“当前状态先于历史事件”的顺序。

```text
【头部·最高权重底线】A1 静态锚点
【头部后段·高权重长期约束】A2 稳定画像 / 关键边界
【中部·补充信息】C 历史事件检索结果
【近尾部·当前局面核心】B 当前工作状态层
【紧邻输入·上下文连续性】D 最近对话窗口
【最尾部·最高即时权重】用户当前输入
```

### 工程实现上的建议表达

```text
System Block 1：静态锚点 + 硬边界
System Block 2：稳定画像 / 长期禁忌
System Block 3：历史事件检索结果
System Block 4：当前工作状态 + 冲突优先级规则
History Messages：最近对话窗口
Human Message：用户当前输入
```

### 当前状态注入原则

- 存储用 JSON
- 注入给模型时转成短摘要
- 明确冲突规则：`硬边界 > 当前状态 > 历史事件 > 近程模糊暗示`
- 不默认把全部元数据直接注入 Prompt

### 5.0 版本语义与阶段边界澄清

为避免“V1 / V2”与“阶段 4 / 5 / 6”互相混淆，本计划将版本语义拆成三类：

#### A2 / B / C 多层候选分流决策表（第一版）

很多真实对话会让同一段文本同时看起来像“长期边界”“当前状态”或“历史事件”。第一版不允许这三层自由竞争，而是采用以下分流优先级：

| 输入特征 | 优先去向 | 可否同时写入其他层 | 说明 |
|----------|----------|--------------------|------|
| 用户明确说“不要这样叫我 / 不要聊这个 / 不要这样回应我” | A2 | 可同步生成 B 层短期状态，但 A2 为主 | 这是高优先级边界，不应只当作当前情绪或普通事件 |
| 用户表达稳定偏好，如“我更喜欢你直接一点” | A2 候选 | 可同步生成 C 事件；不默认直接写 B | 若证据不足，先保留为 A2 候选 + 事件，不急着常驻 |
| 用户当前情绪明显变化，如“我现在很难过 / 放松多了” | B | 可同步生成 C 事件（若具备事件意义） | 情绪变化首先服务于当前回复，不应先写成长期画像 |
| 用户提到一段过去经历，但没有表达长期偏好或当前状态变化 | C | 不默认写 A2 / B | 纯历史信息应优先留在事件层 |
| 用户说“上次那件事后来解决了” | C | 若同时完成了承诺事项，可更新 B 的 `unfinished_items` | 这是事件续写，B 只处理待完成状态的收尾 |
| 用户说“以后别再提这个了，我每次都会难受” | A2 | 可同步写 C；必要时同步写 B 的情绪状态 | 长期禁忌优先，其次才是当前情绪与历史背景 |
| 用户说“这两天我都在焦虑考试” | B | 可同步写 C | 当前焦点 + 情绪为主，是否进入长期画像需等待重复证据 |
| 用户说“我一直都不喜欢别人催我” | A2 候选 | 可同步写 C | 带有长期性表达，但第一版仍按 A2 规则验证稳定性 |
| AI 做出明确承诺，如“明天我再陪你看结果” | B | 可同步写 C | 先写 `unfinished_items`，再决定是否保留为历史事件 |
| 一次性抱怨或上下文拒绝，如“今天不想聊这个” | B 或忽略 | 不进入 A2，必要时可写 C | 单轮有效的信息不应误升为长期边界 |

#### 分流总原则（第一版）

1. **先判 A2，再判 B，最后判 C**
   - 只要命中高优先级边界规则，优先进入 A2。
   - 未命中 A2 时，再判断是否属于当前有效状态。
   - 既不是 A2 也不是当前状态的内容，再进入历史事件层。

2. **A2 与 C 可以并存，但 A2 与 B 的职责不能混淆**
   - A2 表示“以后持续有效的约束或画像”。
   - B 表示“当前几轮内影响回复方式的局面”。
   - C 表示“过去发生过、未来按需回忆的信息”。

3. **同一信息允许主层 + 辅层，不允许三层全写满**
   - 例如“不要叫我宝宝”可以主写 A2，辅写 C 作为来源事件。
   - 但不应同时在 A2、B、C 都注入完整版本，避免重复污染 Prompt。

4. **B 层优先服务当前回复，不承担长期存档职责**
   - 即使某个状态后来会升到 A2，第一步仍应先解决当前是否影响这一轮回应。
   - 但若信息本质上是长期边界，不能只停留在 B 层。

5. **C 层是兜底层，不是垃圾桶**
   - 不能因为不确定就全部丢到 C。
   - C 只保留有事件意义、未来可能需要按需回忆的内容。

6. **推荐实现顺序**
   - 先做 rule-based 分流
   - 再做候选去重
   - 最后做各层写回
   - 不建议先统一落库后再二次猜测归属，否则容易出现多层重复

7. **推荐调试输出**
   - 每个候选至少记录：`candidate_type`、`target_layer`、`why_routed`、`source_round`
   - 阶段 3 调试时应能看到“为什么进入 A2 / B / C”，否则后续很难解释误判

**状态覆盖原则：**
不是只看 `update_time`，而是综合：
- 时间
- `confidence`
- 字段类型优先级
- 是否过期

**状态覆盖逻辑（第一版建议）：**
- 若旧状态已过期，新状态可直接覆盖
- 若新旧字段值相同，仅刷新 `last_update` / 过期计数，不重复制造状态波动
- 若新状态 `confidence` 明显高于旧状态，则允许覆盖
- 若字段优先级更高（如 `relation_state` 高于 `interaction_mode`），高优先级字段不应被低优先级弱信号冲掉
- `unfinished_items` 采用按项合并，而不是整列表覆盖
- 来自 summary 的状态候选默认高于单轮 runtime 弱信号，但不应覆盖明确的近期强信号

#### 状态覆盖阈值口径（第一版固定）

1. **confidence 差值规则**
   - “明显高于”默认指新候选 `confidence - old_confidence >= 0.15`。
   - 若差值 `< 0.15`，默认不因单次弱证据覆盖已有状态，除非旧状态已接近过期或新候选来源等级更高。

2. **来源优先级规则**
   - 默认优先级：`user_explicit / strong_runtime_signal > summary_structured > weak_runtime_rule`。
   - `summary_structured` 可以覆盖 `weak_runtime_rule`，但不应覆盖来自最近 1~2 轮的明确强信号。
   - `user_explicit` 直接触发的状态或边界候选，优先级高于推断类来源。

3. **字段级优先级规则**
   - 字段自身没有互相覆盖关系，优先级主要用于**同字段内部**的覆盖判断，而不是拿 `relation_state` 去覆盖 `interaction_mode`。
   - 第一版同字段覆盖时统一看：是否过期 > 来源等级 > `confidence` > 最近更新时间。

4. **近期强信号定义**
   - “近期强信号”默认指最近 **1~2 个已闭合 round** 内，由用户明确表达或高置信 runtime 分类得到的结果。
   - 若 summary 候选与近期强信号冲突，默认保留近期强信号，直到后续 batch 再次出现同向证据。

5. **`unfinished_items` 合并规则**
   - 先按 `content` 语义相似度做合并；第一版可先采用关键词重叠 + 文本相似的保守规则。
   - `status=done/cancelled` 的项可关闭旧项，不再继续注入；`status=pending` 优先保留。
   - 同时注入数量最多 2 条，其余只保存在状态存储中。

#### 3.7 无变化判断规则（第一版）

阶段 3.7 的目标不是“完全不更新”，而是避免把同义重复、弱波动和无新增证据误当成新记忆。

1. **状态候选的无变化判断**
   - 若同字段的新旧 `value` 相同，且 `confidence` 差值 `< 0.1`，视为无变化，只刷新时间或过期计数。
   - 若 `value` 不同但仅是近义改写，且未改变摘要注入效果，第一版可视为无变化。
   - 若只有 `update_reason` 变化，而用户局面未变，不应视为新状态。

2. **A2 候选的无变化判断**
   - 若候选表达的约束或偏好与现有 A2 语义等价，仅补充证据，不新增新条目。
   - 若只是措辞更强或更弱，但核心边界未变，默认视为同一 A2 项更新。
   - 若来源不同（如从 `summary_inference` 变成 `user_explicit`），可视为证据增强，但不视为新增内容。

3. **历史事件候选的无变化判断**
   - 若事件命中前文去重窗口，且新候选没有新增关键信息，则视为无变化，不新增事件。
   - 若只是重复讲述同一事实，没有结果、进展、承诺变化、冲突升级或状态转折，不应生成新事件。
   - 若新增了结果、反转、时间推进或承诺状态变化，则不属于无变化，可做续写合并或新事件写入。

4. **对 Prompt 注入的无变化判断**
   - 即使底层对象略有更新，只要最终 A2 / B / C 注入摘要与上一轮一致，也可视为“对模型可见层无变化”。
   - 第一版建议优先减少对模型可见层的抖动，而不是追求存储层每次都精细更新。

5. **推荐实现顺序**
   - 先判语义是否变化
   - 再判是否需要更新存储
   - 最后判是否需要更新 Prompt 注入内容
   - 不建议仅凭原始提取文本不同，就直接视为“有变化”

6. **推荐调试输出**
   - `change_type`: `no_change` / `refresh_only` / `content_update` / `new_entry`
   - `why_no_change`: 简短说明，如“same_value_low_delta”“same_boundary_new_evidence”“duplicate_event_no_new_fact”

7. **第一版工程目标**
   - 状态层减少无意义抖动
   - A2 层减少同义重复条目
   - 事件层减少重复入库
   - Prompt 层减少重复注入和表述震荡

**关键文件：**
- `app/service/chat/memory_summary_service.py`
- `app/core/prompts/memory_event_extractor.md`
- `app/core/prompts/current_state_extractor.md`

### 阶段 4：建设历史事件检索层（先做 History Retrieval V1）

**目标：** 在当前简单 Chroma 检索基础上，先建立“面向结构化事件”的 History Retrieval V1，并与 time node 的到期激活形成互补。

**边界说明：**
- 本阶段主要改“检索实现”和“检索结果的内容来源”
- 本阶段不重组整体 Prompt 结构
- 本阶段不新增独立的 `{event_retrieval}` Prompt 字段，默认继续复用现有 `relevant_memories` 注入口
- 本阶段的重点是：让 `relevant_memories` 不再主要承载普通总结文本，而逐步承载“结构化事件检索结果格式化后的文本”
- 若后续需要新增字段或改变层级顺序，应视为 Prompt Layout 调整，归入阶段 6 处理

**任务清单：**

| 序号 | 任务 | 产出物 |
|------|------|--------|
| 4.1 | 扩展 Chroma metadata 以支持事件字段 | 向量库扩展 |
| 4.2 | 实现 `retrieve_history_events()` 或等价检索函数 | 检索函数 |
| 4.3 | 加入去重与预算控制 | 检索后处理 |
| 4.4 | 设计软触发检索逻辑 | 触发规则 |
| 4.5 | 将结构化事件检索结果接到现有 `relevant_memories` 输出契约上 | 接线逻辑 |
| 4.6 | 新增 `format_history_events()` 或等价格式化函数 | 格式化逻辑 |

#### History Retrieval V1 检索策略

- 保留当前向量检索作为基础能力
- 增加最基本的 metadata 过滤
- 增加结果去重
- 控制返回 2~4 条
- 优先返回高 importance / 高相关度 / 新近事件

**事件去重逻辑（第一版建议）：**
- 先按 `actor + type/event_type` 分桶
- 在时间邻近窗口内比较 `content` 与 `keywords` 的相似度
- 对高度相似的候选，仅保留信息更完整、importance 更高、时间更新的一条
- 对“同一事件的多轮延续描述”，允许合并为一条主事件 + 更新后的 metadata，而不是机械重复入库
- 明确禁止仅凭 `event_id` 判断唯一性，因为提取链路重跑时 `event_id` 可能变化

#### 时间邻近窗口与阈值（第一版固定口径）

1. **分桶前提**
   - 只在同一 `actor + event_type` 桶内做去重比较。
   - 不同主体、不同事件类型默认不去重，避免误合并。

2. **短窗重复判定**
   - 若两条候选发生在 **6 轮内** 或 **2 小时内**，进入短窗重复比较。
   - 当 `content` 相似度 `>= 0.85` 且 `keywords` 重叠率 `>= 50%` 时，视为重复事件。
   - 命中重复后，仅保留信息更完整、`importance` 更高、`source_snippet` 更清晰的一条。

3. **长窗续写判定**
   - 若两条候选发生在 **30 轮内** 或 **7 天内**，且第二条包含明显进展信号（如“后来”“结果”“还是”“已经”“最后”），则允许判定为同一事件的续写。
   - 命中续写时，不新建独立事件，而是合并为主事件并刷新 metadata，例如 `last_seen_at`、`source_round`、`source_snippet`、`activity_score`。

4. **超窗策略**
   - 超出长窗后默认视为新事件，不再强行合并。
   - 即使关键词相似，只要时间跨度过大且没有明确续写信号，也按新事件处理。

5. **第一版实现取舍**
   - 若工程复杂度受限，优先实现“短窗重复判定”，把“长窗续写合并”放到后续增强版本。
   - 第一版宁可少合并，也不要因过度激进去重而丢失重要事件。

6. **冲突处理原则**
   - 若相似度接近阈值但不稳定，不直接去重，可同时保留并在检索层再做排序压缩。
   - 去重目标是消除明显重复，不是把相近事件强行压成一条。

7. **推荐保留依据**
   - 优先保留：`importance` 更高 > `source_snippet` 更具体 > 时间更新 > `activity_score` 更高。
   - 若新旧候选信息互补，可保留主事件并补充缺失 metadata，而不是简单覆盖文本。

8. **后续可升级项（非第一版硬要求）**
   - 将 `content` 相似度从纯文本相似扩展为 embedding 相似 + 关键词重叠混合判断。
   - 对不同 `event_type` 使用不同窗口，例如承诺类事件可用更长续写窗口，普通闲聊事件用更短窗口。

#### 触发原则

历史检索的触发不能只看语义回忆词，还要与 time node 的到期激活分工明确。

历史检索的语义触发主要结合：
- 当前输入
- 当前焦点
- 最近消息关键词
- 必要时关系/场景线索
- 已到期 time node 提供的强触发信号

**触发逻辑（第一版建议）：**
- 若当前输入出现明显回忆信号（如“上次”“之前”“你还记得”“后来呢”），直接触发历史检索
- 若当前输入与 `current_focus` 或 `unfinished_items` 高度相关，也触发历史检索
- 若某个 time node 已到期，且其本质更像“需要回忆背景后再继续”的旧事件，可将其作为历史检索的强触发信号
- 若最近消息与当前输入共享较强关键词，但最近窗口本身不足以解释语义跳转，则触发历史检索
- 若当前轮只是普通短句跟答、寒暄或局部接话，且没有回忆信号或 time node 到期信号，默认不触发检索，避免无效召回
- 第一版宁可保守触发，也不要把每轮都变成固定检索，避免时延和噪音上升

#### 历史检索触发阈值与输出契约（第一版）

1. **直接触发条件**
   - 当前输入命中明显回忆词：如“上次”“之前”“后来”“你还记得”“前面说过”。
   - 当前输入直接询问承诺进展、旧事件结果、或对未完成事项做续问。

2. **相关性触发条件**
   - 当前输入与 `current_focus` 文本相似度较高，或关键词重叠明显时触发；第一版可先采用保守规则：关键词重叠 `>= 2` 且当前窗口无法独立解释语义跳转。
   - `unfinished_items` 若存在 `pending` 项，且当前输入命中其核心关键词，也直接触发。

3. **默认不触发条件**
   - 纯寒暄、纯应答、单句情绪反馈、无回忆信号且当前窗口已足够解释上下文时，不触发。
   - 若最近 1~2 轮已刚做过历史检索且当前输入无新增回忆线索，默认不重复触发，避免抖动和重复注入。

4. **输出契约**
   - `format_history_events()` 输出固定为结构化短文本块，不输出原始 JSON。
   - 默认返回 **2~4 条**，每条控制在 **2 行以内**：一行事件本体 + 可选一行时间/进展提示。
   - 每条优先保留：事件内容、主体、必要时的时间或进展；不默认注入全部 metadata。
   - 历史事件总预算以 **300~600 tokens** 为上限；超过预算时，优先裁剪低相关、低 importance、与最近窗口重复的信息。

5. **与最近窗口去重的裁剪规则**
   - 若某事件内容已经在 D 最近对话窗口中直接出现，且窗口文本已足够表达该信息，则不再在 C 层重复注入完整事件。
   - 若历史事件提供的是“背景”或“更早的进展信息”，即使与最近窗口有关键词重叠，也可以保留简化版摘要。

6. **推荐输出样式**

```text
【相关历史事件】
- 用户前几天提到考试压力很大，一直担心结果。
- 之前约好今天继续跟进考试结果，目前仍是待确认状态。
```

7. **第一版实现原则**
   - 先保证“该触发时能触发，不该触发时少打扰”。
   - 先保证输出短、稳、少重复，再追求召回复杂度。
   - 若触发判断无法高置信成立，宁可不触发，也不要把检索变成每轮固定动作。

8. **阶段 7 推荐验收口径**
   - 对“你记得吗 / 后来呢 / 之前说过的那件事”类输入，应能稳定触发历史检索。
   - 对普通寒暄和短句接话，应大多数情况下不触发历史检索。
   - 检索结果不应与最近窗口形成大段重复注入。

#### 本阶段对 Prompt 的影响范围

- 不新增新的整体 Prompt 层级
- 不调整 A1 / A2 / B / C / D 的最终顺序
- 只修改 `relevant_memories` 的内容来源与格式化方式

**关键文件：**
- `app/util/chroma_util.py`
- `app/service/chat_msg_service.py`

### 阶段 7：验证、对比与回滚准备

**目标：** 确保新方案先把正确性、跨日跟进和边界稳定性跑通，再评估 Token 与复杂度收口。

**任务清单：**

**关键验证场景：**

| 场景 | 验证点 |
|------|--------|
| 普通闲聊 | 当前状态不过度污染输出 |
| 连续安慰 / 陪伴 | 状态层持续反映当前局面 |
| 有承诺的对话 | unfinished_items 能生效 |
| 跨日跟进 | time node 到期后能正确激活 |
| 冲突后缓和 | relation_state 能更新 |
| 用户问“你记得吗” | 历史事件召回正确 |
| 提及禁忌话题 | A2 常驻边界能优先生效 |

#### 阶段 7 验收标准（第一版可执行口径）

1. **普通闲聊**
   - 在无明显状态变化、无回忆信号的样本中，B 层不得持续注入与当前轮无关的旧状态。
   - 历史检索默认不触发，或即使触发也不能产生与当前输入无关的大段历史信息。

2. **连续安慰 / 陪伴**
   - 连续数轮围绕同一情绪主题时，`emotion` 与 `interaction_mode` 应保持基本一致，不应每轮抖动。
   - 若主题未变，`current_focus` 应能延续；若主题明显切换，应在合理轮次内更新。

3. **有承诺的对话**
   - 当 AI 明确承诺“下次继续 / 明天再跟进 / 后面再看”后，`unfinished_items` 应在后续相关轮次中仍可被召回。
   - 当事项已完成、取消或明确结束后，该项应停止注入，避免长期悬挂。

4. **跨日跟进**
   - 当用户前一日提到“明天考试 / 下周出结果 / 明天继续聊”这类未来事项后，到期日的新一轮对话中应能稳定激活相关节点。
   - 即使用户本轮输入没有显式重复旧话题，系统也应具备“知道今天该想起这件事”的能力。
   - 已完成、已取消或明显过期的节点不应继续触发。

5. **冲突后缓和**
   - 当关系氛围从紧张转向缓和时，`relation_state` 应允许更新，而不是永久停留在旧状态。
   - 但这种更新不应因单轮模糊措辞就频繁反转，至少应符合前文的 confidence / 来源优先级规则。

6. **用户问“你记得吗”**
   - 对明确回忆型输入，应稳定触发历史检索。
   - 返回的历史事件应与问题直接相关，不应主要返回语义近但无用的旧摘要。
   - 若相关信息已在最近窗口中充分出现，则 C 层应避免重复注入整段历史。

7. **提及禁忌话题**
   - 命中 A2 高优先级边界时，A2 约束必须优先于 B 层状态和 C 层历史事件生效。
   - 不得出现“明知是禁忌仍因为历史事件召回而继续展开”的情况。

8. **Token 与重复控制**
   - 第一版优先看 correctness，不以 Token 下降作为通过前提。
   - 相比阶段 0 基线，动态注入部分允许短期内不降，甚至小幅上升，但必须可解释。
   - 不应出现同一信息同时在 A2、B、C、D 四层中大段重复注入。

9. **最低验收方式**
   - 阶段 7 至少对每个关键场景准备 1 个可复现实例。
   - 每个实例都要记录：输入片段、实际注入块、模型输出、人工结论。
   - 若某项未通过，应能定位是路由问题、检索问题、状态覆盖问题、time node 触发问题还是 Prompt 表达问题。

10. **第一版通过原则**
   - 第一版不追求所有场景全自动满分，但至少要做到：主要场景可复现、问题可归因、回滚可执行。
   - 若 Token 略有增加但状态一致性、跨日跟进能力和边界正确性显著提升，可接受进入小范围对比阶段；若边界、承诺或 time node 跨日跟进场景仍频繁失败，则不应视为通过。

11. **与阶段 0 的衔接要求**
   - 阶段 7 的评估样本应尽量复用阶段 0 的样本集，保证前后对比口径一致。
   - 对时间相关能力，必须复用阶段 0 构造的时间样本集，避免因为样本漂移导致误判。
   - 若新增特殊样本，必须在报告中说明新增原因，避免“换题后看起来变好了”的假象。

12. **推荐报告结构**
   - `新旧架构对比结论`
   - `按场景的通过/失败列表`
   - `失败原因归类`
   - `是否需要回滚或继续灰度`

13. **不通过的硬条件**
   - A2 高优先级边界失效
   - 承诺事项无法稳定跟进
   - time node 到期后仍无法稳定激活
   - 历史检索大面积误召回
   - 动态注入重复明显增加且无法解释
   - 无法通过开关回退到旧架构

14. **推荐通过门槛（第一版）**
   - 关键验证场景中，不应出现 A2 边界类硬失败。
   - 历史检索相关场景应达到“多数样本判断为正确召回”。
   - 当前状态相关场景应达到“多数样本判断为状态一致且不过度污染”。
   - time node 相关样本应达到“多数样本判断为到期后可正确激活，不到期时不误触发”。
   - 由于第一版仍是演进式改造，若偶发样本表现不稳定，但总体趋势优于旧架构，可进入下一轮优化而不是立刻否决。

15. **推荐判定记录格式**
   - `场景类型`：如“承诺跟进”
   - `是否通过`：通过 / 不通过 / 待复核
   - `主要问题层`：A2 / B / T / C / D / Prompt Layout
   - `证据摘要`：1~2 句说明
   - `是否可回归复现`：是 / 否

16. **推荐附加指标（可选）**
   - 检索触发率
   - 触发后有效率
   - time node 激活正确率
   - time node 误触发率
   - 状态覆盖稳定率
   - 多层重复率
   - 边界命中正确率

这些验收标准的目标不是把阶段 7 变成大而全的测试平台，而是确保这次架构升级至少具备“可比较、可归因、可回退”的工程落地条件。

## 七、风险与回滚

### 7.1 核心风险

| 风险 | 应对措施 |
|------|----------|
| 检索召回率下降 | 保留旧架构开关，支持回退 |
| 状态层过于复杂 | 先做最小字段集合，再逐步扩展 |
| 摘要与事件检索污染 | 拆 collection 或严格 metadata 隔离 |
| LLM 提取失败 | 保留原状态 / 旧 memory_bank 作为过渡保底 |
| Token 未下降 | 检查 recent window、检索数量、重复信息消减 |

### 7.2 回滚方案

- 保留旧 memory_bank 注入逻辑作为旧架构 fallback
- 保留旧 Prompt 结构切换开关
- 新旧架构在一段时间内支持并行对比

## 八、实施顺序建议

```text
Phase 1：基线与契约
  - 阶段 0 + 阶段 1
  - 先知道现状，再把 A1 / A2 / B / T / C / D 的对象和边界定清楚

Phase 2：状态层与时间节点落地
  - 阶段 2
  - 先把 emotion_state 升级成 current_state 容器，并补上最小 time node 运行时能力

Phase 3：事件提取与路由
  - 阶段 3
  - 让总结流程分别产出 A2、状态、time node 和事件候选

Phase 4：历史检索层 V1
  - 阶段 4
  - 在现有 Chroma 基础上先跑通结构化事件检索，并与 time node 到期激活联动

Phase 5：Prompt 重组
  - 阶段 6
  - 按最终顺序重排注入结构

Phase 6：验证与回滚准备
  - 阶段 7

Phase 7：检索增强（可选 V2）
  - 阶段 5
  - 在 V1 架构成立后，再升级混合检索 / rerank / 类型化衰减
```

这个顺序比“先大改流程，再补状态契约”更稳，因为它先把最关键的对象和边界定住。
