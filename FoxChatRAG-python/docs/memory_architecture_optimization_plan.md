## 二、当前项目现状判断

### 2.1 当前已经相对成熟的部分：永久锚点层

当前系统中，下列内容已经具备“长期稳定、每轮应当注入”的特征：

- 角色灵魂（Soul）
- 角色声明 / 核心锚点（Core Anchor）
- 角色卡（Character Card）
- 用户画像（User Profile）

这些内容的主要作用不是回忆具体往事，而是：

- 固定角色身份
- 固定边界与说话风格
- 固定角色对用户的总体认识
- 防止模型偏离设定

因此，本文将其统一归为“永久锚点层”，默认保留，不作为本轮优化重点。

### 2.2 当前主要问题不在永久锚点，而在动态层

目前的主要问题集中在两部分：

#### 问题 A：历史记忆的注入方式不合理

当前 Memory Bank 以 JSON 数组形式存储在 Redis 中，并在对话时被整体展开为文本注入提示词。

这会带来三个直接问题：

1. 无关记忆也被一并注入，浪费 Token
2. 记忆越积越多，动态部分成本不断上升
3. 记忆的“相关性选择”能力弱，无法只取当前真正需要的历史信息

#### 问题 B：当前状态层过于单薄

项目中已经实现了“情绪状态”机制，但目前的情绪状态只覆盖：

- 当前情绪标签
- 确定性
- 最后更新时间
- 触发情绪变化的片段

它只能回答“角色现在大概是什么情绪”，但不能完整回答：

- 角色与用户当前关系处于什么阶段
- 当前是否有持续中的承诺、约定、任务、冲突
- 当前话题是否仍在延续
- 当前对话的处境是什么
- 哪些状态是“现在有效”的，而不是过去发生过的

因此，当前情绪状态可以保留，但不能替代完整的“当前状态层”。

### 2.3 一个关键现实：长期检索层实际上还没有真正接管主流程

从当前代码逻辑看，系统虽然保留了“相关记忆”这一注入口，但主对话流程仍主要依赖：

- 最近消息
- 永久锚点
- Redis 中整体展开的 Memory Bank

也就是说，现阶段真正消耗上下文的大头，仍然是“动态信息的整体注入”，而不是“按需检索后的精简注入”。

这意味着本轮优化的本质，不是简单换存储位置，而是要完成一次**动态记忆职责重构**。

## 三、目标架构：三层记忆注入模型

### 3.1 总体思路

目标架构拟分为三层：

```text
┌─────────────────────────────────────────────┐
│                三层记忆注入模型              │
├─────────────────────────────────────────────┤
│  第一层：永久锚点层                          │
│  - Soul / Core Anchor / Character Card      │
│  - User Profile                             │
│  作用：固定角色身份、风格、边界、稳定认知      │
├─────────────────────────────────────────────┤
│  第二层：历史事件检索层                      │
│  - 已发生的重要事件                          │
│  - 已形成的阶段性关系变化                    │
│  - 过去承诺、冲突、共同经历                  │
│  作用：按需回忆，支撑“记得过去发生过什么”      │
├─────────────────────────────────────────────┤
│  第三层：当前状态层                          │
│  - 最近消息                                  │
│  - 当前情绪                                  │
│  - 当前关系态势                              │
│  - 当前持续中的话题/承诺/任务/冲突            │
│  作用：支撑“现在正在发生什么、该如何接着说”    │
└─────────────────────────────────────────────┘
```

### 3.2 各层职责划分

#### 第一层：永久锚点层

职责：

- 定义角色是谁
- 定义角色不能越过什么边界
- 定义用户在长期视角下的大致画像
- 保证模型不轻易偏离角色设定

特点：

- 稳定
- 更新频率低
- 每轮常驻注入
- 不依赖检索

#### 第二层：历史事件检索层

职责：

- 回忆过去发生过的重要事情
- 回忆过去形成过的互动节点
- 为“你还记得吗”“上次说过”“之前答应过”这类场景提供依据

特点：

- 不应全量注入
- 应按需召回
- 以“事件”为主，而不是以“状态”为主
- 必须支持去重、版本判断、相关性排序

#### 第三层：当前状态层

职责：

- 告诉模型当前局面是什么
- 告诉模型哪些状态现在仍然有效
- 弥补“最近消息窗口有限”导致的上下文断裂

特点：

- 小而精
- 每轮常驻注入
- 表达“当前有效状态”，而不是“过去曾经发生过”
- 可由最近消息、情绪系统、阶段总结共同更新

## 四、为什么“情绪状态”不能替代完整状态层

### 4.1 当前情绪状态是有价值的，但它只是一维状态

当前情绪状态有明确意义：

- 它能给角色回复增加连续性
- 它能让角色不像每轮都重置
- 它适合作为当前状态层中的一个字段

但它本质上只回答一个问题：

> 角色现在大概是什么情绪？

而完整状态层至少还要回答以下问题：

- 角色现在和用户是亲近、紧张、疏远、依赖还是试探
- 当前有没有未完成的话题或承诺
- 当前是不是处在某个连续场景中
- 当前这轮回复应该延续什么互动方向
- 某些边界、称呼、立场是否在当前阶段有特殊强化

如果没有这些信息，模型即使知道“当前情绪是开心”，也仍然可能：

- 忘记当前正在谈什么
- 忘记刚形成的默契或冲突
- 忘记某件事现在仍在推进中
- 在语义上表现得“有情绪，但没处境”

### 4.2 因此，建议保留“状态层”，但缩小其范围并明确职责

建议将“当前状态层”定义为：

- 一个轻量、常驻、表达当前有效信息的对象
- 情绪状态只是其中一个字段
- 不把整个长期历史混进来
- 不负责记录所有往事

换言之：

- **情绪状态应保留**
- **但情绪状态不等于完整状态层**
- **结构化记忆中仍然需要保留“状态”概念，但应与历史事件严格区分**

***

## 五、目标数据观：事件和状态必须分开看待

### 5.1 事件是什么

事件是“已经发生过”的内容，例如：

- 用户提到自己最近要考试
- 角色答应明天继续聊某个话题
- 用户上次表达了失落
- 双方在某轮对话里产生了冲突或达成了默契

事件的特点：

- 有时间性
- 有参与者
- 有内容
- 可以进入历史检索层

### 5.2 状态是什么

状态是“当前仍然有效”的内容，例如：

- 当前情绪：平静 / 开心 / 紧张
- 当前关系态势：亲近 / 试探 / 疏离 / 依赖
- 当前持续任务：正在安慰、正在争论、正在陪伴、正在确认计划
- 当前场景：刚发生冲突后、安抚阶段、轻松闲聊阶段
- 当前对话焦点：考试、休息、承诺、边界、约定

状态的特点：

- 不是单次事件
- 强调“现在有效”
- 会被新状态覆盖或更新
- 不适合完全依赖向量检索决定

### 5.3 设计原则

因此，本方案建议：

- 历史事件进入检索层
- 当前状态进入常驻层
- 状态变化可以来源于事件，但不能等同于事件
- 事件可以被检索回来补充历史背景，状态必须有“当前版本”

***

## 六、建议中的结构化字段方向

本节不写代码，只定义字段思路，供老师讨论。

### 6.1 历史事件对象建议字段

建议历史事件至少具备以下信息：

| 字段 | 用途 |
|------|------|
| 事件 ID | 便于去重、更新、追踪 |
| 时间 | 判断先后关系和新旧程度 |
| 参与者 | 明确是谁做了什么 |
| 事件类型 | 区分承诺、冲突、偏好表达、共同经历、边界触发等 |
| 事件内容 | 事件本体 |
| 关键词 | 支持关键词召回与检索过滤 |
| 重要度 | 控制召回优先级 |
| 来源轮次 / 来源窗口 | 便于回溯与调试 |
| 是否仍相关 | 防止陈旧事件长期污染检索 |

### 6.2 当前状态对象建议字段

建议当前状态至少具备以下信息：

| 字段 | 用途 |
|------|------|
| 当前情绪 | 表达角色情绪连续性 |
| 当前关系态势 | 表达互动关系当前处于什么阶段 |
| 当前焦点话题 | 告诉模型此刻主要围绕什么展开 |
| 当前未完成事项 | 承诺、计划、等待回应的事情 |
| 当前互动模式 | 安慰、陪伴、争论、试探、闲聊等 |
| 生效时间 | 判断状态是否过期 |
| 最后更新时间 | 便于替换旧状态 |
| 触发依据摘要 | 便于调试为何进入该状态 |

### 6.3 对结构化记忆的总体要求

下一阶段的结构化记忆必须满足：

1. 能区分“过去发生的事”和“现在仍有效的状态”
2. 能区分“角色说的”和“用户说的”
3. 能区分“旧状态”和“新状态”
4. 能支持检索、排序、去重和调试

如果做不到这些，检索回来的信息就容易：

- 主体错乱
- 新旧冲突
- 重复堆积
- 与当前语境不符

## 七、从当前项目到目标架构的优化计划

以下计划按“先拆职责，再改结构，再接检索，最后验证”的顺序设计。

### 阶段 0：建立基线与评估标准

**目标：** 在改架构之前，先知道当前系统到底哪里最贵、哪里最容易错。

**需要完成的工作：**

- 记录当前每轮提示词注入的组成与 Token 占比
- 记录最近消息、Memory Bank、情绪状态各自的体量
- 收集若干真实对话样本，标注当前错误类型
- 明确本轮优化的评估指标

**建议重点观测的指标：**

- 每轮总 Token
- 动态部分 Token 占比
- 历史事件召回正确率
- 当前状态一致性
- 角色设定稳定性
- 响应时延变化

**产出物：**

- 当前系统基线报告
- 评测样本集合
- 优化目标阈值

---

### 阶段 1：先拆开“历史事件”和“当前状态”的职责

**目标：** 停止把动态记忆当成一个大桶看待。

**需要完成的工作：**

- 重新定义哪些内容属于永久锚点
- 重新定义哪些内容属于历史事件
- 重新定义哪些内容属于当前状态
- 明确三层在提示词中的注入顺序和预算

**关键原则：**

- 永久锚点负责“你是谁”
- 历史事件负责“以前发生过什么”
- 当前状态负责“现在处于什么局面”

**产出物：**

- 三层职责表
- 注入预算初稿
- 状态与事件的边界定义

---

### 阶段 2：重构结构化记忆的数据模型

**目标：** 让记忆对象可检索、可去重、可判断新旧。

**需要完成的工作：**

- 细化事件对象字段
- 细化当前状态对象字段
- 定义事件类型分类
- 定义状态覆盖规则
- 定义重复事件的合并原则

**本阶段核心问题：**

- 什么算一条独立事件
- 什么算一个状态变化
- 多轮总结中重复提取到相似事件时如何处理
- 旧状态是否保留为历史，还是只保留最新态

**产出物：**

- 事件对象草案
- 状态对象草案
- 去重与覆盖规则文档

---

### 阶段 3：重构记忆生成流程

**目标：** 让系统不再只会“总结成一坨”，而是能分别产出事件和状态。

**需要完成的工作：**

- 重新设计总结后的产出形式
- 让总结流程分别生成：
  - 历史事件候选
  - 当前状态候选
- 设计事件入库前的筛选逻辑
- 设计状态更新前的覆盖逻辑

**本阶段重点：**

- 不是每次总结都必须产生状态变更
- 不是每条对话都值得成为历史事件
- 事件和状态都应允许“无变化”

**产出物：**

- 新的记忆生成流程图
- 事件筛选规则
- 状态更新规则

---

### 阶段 4：建设历史事件检索层

**目标：** 用检索替代全量注入，让长期记忆按需进入上下文。

**需要完成的工作：**

- 为历史事件建立独立检索单元
- 明确每次检索返回多少条、允许多长
- 设计检索排序策略
- 定义何时触发历史检索
- 定义检索失败时的行为

**建议原则：**

- 以事件检索为主，不以状态检索为主
- 检索数量必须有上限
- 检索结果应优先保证正确，而不是保证多
- 检索层必须可调试，可知道为什么召回了某条记忆

**本阶段要特别防范的风险：**

- 检索到旧状态冒充当前状态
- 检索到相似但不相关的事件
- 检索结果重复
- 事件块过大，导致检索回来依旧浪费 Token

**产出物：**

- 历史事件检索方案
- 检索排序和过滤策略草案
- 检索预算方案

---

### 阶段 5：建设当前状态层

**目标：** 让模型每轮都知道“当前局面”，而不是只知道“当前情绪”。

**需要完成的工作：**

- 保留现有情绪状态作为状态层一部分
- 增补关系态势、持续任务、当前焦点等字段
- 定义状态更新频率
- 定义状态过期机制
- 定义哪些状态必须常驻注入

**关键判断：**

当前状态层应当：

- 比历史事件层短得多
- 比最近消息更抽象一点
- 比永久锚点更新更频繁
- 比情绪状态更完整

**产出物：**

- 当前状态层字段定义
- 状态刷新策略
- 状态失效规则

---

### 阶段 6：重构提示词注入策略

**目标：** 让三层记忆真正按不同职责注入，而不是继续混在一起。

**需要完成的工作：**

- 调整提示词注入结构
- 明确三层的顺序、长度预算和用途说明
- 避免同一信息在多个层重复出现
- 减少无关历史对当前回复的干扰

**建议注入顺序：**

1. 永久锚点层
2. 当前状态层
3. 历史事件检索结果
4. 最近消息
5. 用户当前输入

**设计理由：**

- 先固定身份与边界
- 再固定当前局面
- 再补过去相关事件
- 最后让模型处理最近上下文和当前输入

**产出物：**

- 新提示词结构说明
- 三层预算表
- 重复信息消减规则

---

### 阶段 7：验证、对比与回滚准备

**目标：** 在正式替换旧架构前，确保新方案不是“省了 Token，却丢了正确性”。

**需要完成的工作：**

- 用相同对话样本对比旧架构与新架构
- 观察角色稳定性是否下降
- 观察历史记忆召回是否准确
- 观察当前状态是否连续
- 预留回滚方案，避免一次性替换全部逻辑

**建议重点验证场景：**

- 普通闲聊
- 连续安慰 / 陪伴类对话
- 有承诺、有约定的对话
- 出现冲突后再缓和的对话
- 用户频繁提及“上次、之前、你记得吗”的对话

**产出物：**

- 新旧方案对比报告
- 风险清单
- 是否进入正式替换的结论

## 八、实施时的核心风险

### 风险 1：把状态也完全检索化

如果把“当前状态”也完全交给向量检索，最容易出现的问题是：

- 旧状态被召回
- 新状态被遗漏
- 当前局面判断失真

因此，状态层必须保留“当前版本”的常驻表示。

### 风险 2：事件块设计过大

如果一个检索块中同时包含太多事件、太多状态、太多背景，虽然上下文完整，但检索精度会下降，Token 也会重新膨胀。

因此，事件块应控制粒度，不宜过大。

### 风险 3：重复事件堆积

当前总结机制具有窗口重叠特征，若没有去重和覆盖规则，极易反复提取相似事件。

因此，事件层建设前，必须先设计好去重与合并规则。

### 风险 4：结构越来越复杂，但收益不明显

如果引入过多字段、过多状态分类、过多检索规则，可能导致：

- 开发复杂度急剧上升
- 调试困难
- 模型输出稳定性反而下降

因此，状态层应坚持“小而稳定”的原则，不应贪多。

***

## 九、本方案中最值得老师提意见的问题

为提高评审效率，建议老师重点审查以下问题：

### 9.1 关于状态层边界

- 当前状态层应当包含哪些字段才算够用
- 情绪、关系态势、持续任务三类中，哪些是必须的，哪些可以后置
- 当前状态是否需要显式过期机制

### 9.2 关于历史事件粒度

- 一条历史事件应当多大为宜
- 是否应当允许“阶段性事件摘要”与“单点事件”并存
- 哪些事件值得长期保留，哪些应当自然淘汰

### 9.3 关于检索策略

- 历史事件应当每轮都检索，还是按触发条件检索
- 检索结果数量如何设上限
- 召回正确率与召回覆盖率，哪个优先级更高

### 9.4 关于状态与事件关系

- 状态是否应从事件中推导，还是独立生成
- 旧状态是否应保留为历史材料
- 某些“关系变化”到底归类为事件还是状态

### 9.5 关于迁移节奏

- 是应当逐层替换，还是一次性整体替换
- 是否需要并行保留旧 Memory Bank 一段时间作为对照
- 验证样本如何选取才能有说服力

***

## 十、当前建议结论

基于目前项目现状，本文建议如下：

### 10.1 保留的部分

- 永久锚点层保留
- 用户画像机制保留
- 角色卡 / 核心锚点机制保留
- 现有情绪状态机制保留，但降级为“当前状态层中的一个字段”

### 10.2 重点重构的部分

- 不再将 Memory Bank 作为整库常驻注入
- 建立独立的历史事件检索层
- 建立小而稳定的当前状态层
- 重构总结流程，使其分别产出事件和状态

### 10.3 最核心的原则

> 永久锚点负责“角色是谁”；  
> 历史事件负责“过去发生过什么”；  
> 当前状态负责“现在正处于什么局面”。

只有这三件事真正拆开，记忆架构才有可能既省 Token，又保持长期一致性。

***

## 十一、建议的下一步

在真正开始实现前，建议先完成以下三项非代码工作：

1. 先确定“当前状态层”的最小字段集合
2. 先确定“历史事件对象”的最小字段集合
3. 先确定检索预算与评估标准

这三件事如果不先定清楚，后续实现很容易再次变成“存了很多结构，但不知道真正该怎么用”。

***

## 十二、一句话总结

本轮优化不应理解为“把记忆从 Redis 挪到向量库”，而应理解为：

> 将当前混杂的动态记忆，拆分为“可检索的历史事件”和“常驻的当前状态”，  
> 在不破坏角色锚定能力的前提下，降低 Token 消耗并提升长期一致性。
