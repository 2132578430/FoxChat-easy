# FoxChat 推荐记忆流程文

> 日期：2026-04-26  
> 用途：用于方案收敛、流程讨论与实现前评审  
> 目标：给出一套当前阶段最推荐的记忆分层、注入顺序、读写流程与实施顺序

***

## 一、本文结论

当前阶段最推荐的方向不是“继续把所有动态记忆塞进一个 Memory Bank”，也不是“直接整体替换成外部记忆系统”，而是：

> 将原本混杂的动态记忆拆成三类职责：  
> **稳定层负责固定身份和长期边界，当前状态层负责表达现在的局面，历史检索层负责回忆过去的重要事件。**

同时，最近几轮原始对话应作为单独的“工作窗口”存在，而不应再被混同为长期记忆。

因此，推荐的整体结构是：

- 常驻稳定层
- 当前工作状态层
- 历史事件检索层
- 最近对话窗口

***

## 三、各层职责说明

### 3.1 A 层：常驻稳定层

常驻稳定层不负责表达“当前场景”，主要负责表达：

- 角色是谁
- 角色不能越过哪些底线
- 用户有哪些长期稳定偏好
- 哪些长期禁忌不能被遗忘

该层又建议分成两部分：

#### A1. 静态锚点

包括：

- soul
- core_anchor
- character_card

特点：

- 基本不变
- 每轮常驻注入
- 权重应当最高
- 主要作用是防止角色偏离

#### A2. 稳定画像 / 关键边界

包括：

- user_profile
- 长期偏好
- 长期禁忌
- 永不衰减的边界信息

特点：

- 会更新，但更新频率低于当前状态层
- 可在总结流程中逐步修正
- 每轮常驻注入
- 不应依赖检索召回才生效

因此，`user_profile` 虽然不是绝对静态，但仍应放在“常驻稳定层”，而不应归入“当前状态层”。

---

### 3.2 B 层：当前工作状态层

当前工作状态层不负责记录所有历史，而负责表达：

> 角色现在正处于什么局面，这一轮应该以什么态度、方向和上下文继续回应。

推荐第一版的最小字段集合为：

- `emotion`
- `relation_state`
- `current_focus`
- `unfinished_items`
- `interaction_mode`

#### 字段含义

##### emotion
角色当前情绪状态。

##### relation_state
角色与用户当前关系气氛，如：亲近、试探、疏离、安抚中、轻微冲突后缓和。

##### current_focus
当前最主要的话题或关注点，如：考试压力、洗漱休息、上次约定、边界确认。

##### unfinished_items
当前未完成的事项，如：等待用户继续说某件事、承诺下轮继续解释、明天继续聊结果。

##### interaction_mode
当前互动模式，如：陪伴、安慰、闲聊、争论、试探、计划确认。

#### 推荐元数据

每个状态字段建议在内部附带以下元数据：

- `confidence`：置信度
- `expire_rounds` 或等价过期机制：防止临时状态长期常驻
- `update_reason`：状态更新原因，主要用于调试与回溯

注意：

- 元数据主要用于存储、调试和更新逻辑
- 不建议原样把全部元数据直接注入 Prompt
- 模型看到的应该是压缩后的状态摘要，而不是原始状态 JSON

---

### 3.3 C 层：历史事件检索层

历史事件检索层只负责一件事：

> 在当前输入和最近上下文的基础上，召回与本轮真正相关的“过去事件”。

它不负责表达“当前什么状态仍有效”，也不应承担所有边界管理职责。

推荐第一版采用结构化事件对象，而不是大块混合记忆。

#### 推荐最小事件字段

- `event_id`
- `time`
- `actor`
- `type`
- `content`
- `keywords`
- `importance`
- `source_snippet`

#### 字段作用

- `time`：判断时间先后与新旧程度
- `actor`：明确是谁说了什么、做了什么
- `type`：区分承诺、冲突、偏好表达、共同经历、边界触发等
- `content`：事件摘要
- `keywords`：辅助关键词召回
- `importance`：控制优先级
- `source_snippet`：保留少量原文，防止摘要失真

#### 推荐增强方向

- 类型化衰减：不同类型事件采用不同衰减策略
- 否定性边界提升：重要禁忌不只留在历史层，而要提升到常驻稳定层
- 混合检索：BM25 + 向量检索
- 轻量 rerank：解决“上次那个”“之前说的”这类指代性召回问题

---

### 3.4 D 层：最近对话窗口

最近对话窗口不属于长期记忆，而属于工作窗口。

它的职责是：

- 保留上一轮和当前轮之间的直接连续性
- 保留代词指向和上下句关系
- 保留当前说话语气、节奏和短程上下文

推荐保留最近 4~6 轮原始消息，而不是无限增长。

## 四、推荐 Prompt 注入顺序

结合角色扮演需求、当前状态优先级，以及 LLM 常见的 U 型注意力规律，推荐采用以下顺序：

```text
【头部·最高权重底线】A1 静态锚点
【头部后段·高权重长期约束】A2 稳定画像 / 关键边界
【中部·补充信息】C 历史事件检索结果
【近尾部·当前局面核心】B 当前工作状态层
【紧邻输入·上下文连续性】D 最近对话窗口
【最尾部·最高即时权重】用户当前输入
```

### 设计理由

#### 1. 静态锚点放最前
因为它定义角色底线，不应被覆盖。

#### 2. 稳定画像 / 关键边界紧随其后
因为长期偏好、长期禁忌也应具有较高权重，不能完全埋在中间。

#### 3. 历史事件放中间
历史事件的作用是“补充过去”，不是压倒当前局面，因此放中部更合适。

#### 4. 当前状态放近尾部
当前状态应尽量接近模型最终生成位置，使模型优先感知“现在该怎么说”。

#### 5. 最近对话紧邻用户输入
这可以最大化连贯性，让模型顺着上一轮自然接下去。

### 工程实现上的建议表达

与其把所有内容塞进一个巨大的系统提示词，不如在概念上按以下顺序组织：

```text
System Block 1：静态锚点 + 硬边界
System Block 2：稳定画像 / 长期禁忌
System Block 3：历史事件检索结果
System Block 4：当前工作状态 + 冲突优先级规则
History Messages：最近对话窗口
Human Message：用户当前输入
```

## 五、状态如何注入

状态注入不只是“放在 Prompt 的哪个位置”，而是四件事同时决定的：

1. 注入什么字段
2. 用什么形式表达
3. 放在什么位置
4. 冲突时谁优先

### 推荐原则

#### 5.1 存储用 JSON，注入用精简摘要

内部状态可以是 JSON，但给模型的最好是压缩后的文本摘要。例如：

```text
【当前状态】
- 当前情绪：安抚中
- 当前关系态势：亲近、信任
- 当前焦点：用户正在倾诉考试失利
- 当前未完成事项：等待用户继续说考试经过
- 当前互动模式：陪伴 / 安慰

规则：
- 若历史事件与当前状态冲突，以当前状态和最近对话为准
- 若当前输入与硬边界冲突，仍需遵守硬边界
```

#### 5.2 状态要短
当前状态层的目标不是替代历史，而是让模型快速知道“此刻局面”。

#### 5.3 状态需要冲突优先级
推荐的优先级原则是：

```text
硬边界 / 禁忌 > 当前状态 > 历史事件 > 最近对话中的模糊暗示
```

#### 5.4 状态要有过期机制
否则临时状态会永久常驻，导致系统与当前语境脱节。

## 六、推荐读流程（用户发来消息时）

```text
用户输入
   │
   ▼
读取 A1 静态锚点
   │
   ├─ soul
   ├─ core_anchor
   └─ character_card
   │
   ▼
读取 A2 稳定画像 / 关键边界
   │
   ├─ user_profile
   ├─ 长期偏好
   └─ 长期禁忌 / 高优先级边界
   │
   ▼
读取 B 当前工作状态层
   │
   ├─ emotion
   ├─ relation_state
   ├─ current_focus
   ├─ unfinished_items
   └─ interaction_mode
   │
   ▼
读取 D 最近对话窗口
   │
   └─ 最近 4~6 轮原始消息
   │
   ▼
生成检索查询
   │
   ├─ 当前输入
   ├─ 当前焦点
   ├─ 最近消息中的关键词
   └─ 必要时加入关系/场景线索
   │
   ▼
从 C 历史事件层召回相关事件
   │
   ├─ 向量检索
   ├─ 关键词检索
   ├─ 过滤冲突项
   ├─ 去重
   └─ 限制总长度
   │
   ▼
组装 Prompt
   │
   ├─ A1 静态锚点
   ├─ A2 稳定画像 / 关键边界
   ├─ C 历史事件检索结果
   ├─ B 当前工作状态层
   ├─ D 最近对话窗口
   └─ 用户当前输入
   │
   ▼
LLM 生成回复
```

## 七、推荐写流程（模型回复之后）

```text
模型回复完成
   │
   ▼
保存本轮原始消息
   │
   ├─ human message
   └─ ai message
   │
   ▼
后台更新情绪状态
   │
   └─ emotion_state 更新
   │
   ▼
判断是否达到记忆整理阈值
   │
   ├─ 否 → 结束
   └─ 是 → 进入整理
   │
   ▼
从旧消息窗口中抽取候选信息
   │
   ├─ 稳定偏好 / 长期边界候选
   ├─ 当前状态变化候选
   └─ 历史事件候选
   │
   ▼
执行信息路由
   │
   ├─ 长期稳定偏好 / 禁忌 → A2 稳定画像 / 关键边界
   ├─ 当前仍然有效的局面   → B 当前工作状态层
   ├─ 过去发生的重要事件   → C 历史事件检索层
   └─ 仅保留局部连贯性     → D 最近对话窗口
   │
   ▼
状态处理
   │
   ├─ 判断是否覆盖旧状态
   ├─ 应用过期规则
   └─ 写回 current_state
   │
   ▼
事件处理
   │
   ├─ 去重
   ├─ 合并相似项
   ├─ 设置 metadata
   └─ 写入检索层
   │
   ▼
稳定画像处理
   │
   ├─ 更新 user_profile
   ├─ 提升高优先级禁忌
   └─ 保持长期边界常驻
   │
   ▼
裁剪最近对话窗口
```

## 八、信息路由规则

本方案的关键不是“继续往 Memory Bank 里加内容”，而是：

> 每次从总结或抽取中得到的信息，都必须先判断属于哪一层。

推荐路由规则如下：

### 8.1 如果是长期稳定偏好 / 长期禁忌
进入 A2 稳定画像 / 关键边界层。

示例：

- 用户怕黑
- 用户不喜欢被某种称呼称呼
- 用户长期喜欢某类互动方式

### 8.2 如果是当前仍然有效的局面
进入 B 当前工作状态层。

示例：

- 当前正在安慰用户
- 当前关系正处于缓和阶段
- 当前主要在讨论考试失利
- 当前有一个未结束的话题等待继续

### 8.3 如果是过去发生的重要事件
进入 C 历史事件检索层。

示例：

- 上次答应继续聊某件事
- 某轮发生过冲突
- 双方形成过一个共同回忆
- 用户提到过一个重要经历

### 8.4 如果只是局部上下句连续性
保留在 D 最近对话窗口，不进入长期记忆。

示例：

- 临时感叹
- 一次性闲聊句子
- 当前轮只对局部句法有意义的信息

## 九、实施顺序建议

### 阶段 1：先定对象契约

在写代码前，先确定三类对象：

- 稳定画像 / 关键边界对象
- 当前状态对象
- 历史事件对象

不先定对象契约，后续所有实现都容易再次混成一个大桶。

---

### 阶段 2：将 emotion_state 升级为 current_state 容器

推荐先复用现有情绪状态的存储思路，把它扩成 `current_state`，逐步加入：

- relation_state
- current_focus
- unfinished_items
- interaction_mode

情绪保留，但不再独占整个状态层。

---

### 阶段 3：建立信息路由机制

这是本方案的核心。

系统不能再把总结结果一股脑全部塞进同一个 Memory Bank，而必须判断信息流向：

- 进 A2
- 进 B
- 进 C
- 或仅留在 D

---

### 阶段 4：跑第一版历史检索层

推荐第一版先采用：

- 结构化历史事件对象
- Chroma 向量检索
- 基础关键词召回

先验证架构是否成立，不急于一开始就引入更重的外部系统。

---

### 阶段 5：在架构稳定后再升级检索

如果第一版的主要问题集中在历史召回不足，则再考虑：

- BM25 + 向量检索混合召回
- rerank
- MemPalace 风格的原文保真历史检索

也就是说，MemPalace 更适合作为未来升级历史检索层的方向，而不是当前直接整体替换现有系统。

***

## 十、推荐预算策略

为了防止架构升级后 Token 再次失控，建议从一开始就设预算：

- A1 静态锚点：固定预算
- A2 稳定画像 / 关键边界：固定预算
- B 当前工作状态层：200~400 tokens
- C 历史事件检索层：2~4 条事件
- D 最近对话窗口：最近 4~6 轮

原则：

- 当前状态层必须短
- 历史检索层必须限量
- 最近消息不能无限增长
- 不允许同一信息在 A/B/C 三层重复表达

***

## 十一、决策门

为了避免系统一次改太多，建议按以下决策门推进：

### Gate 1：当前状态层是否已经稳定？

如果未稳定：
- 不要急着升级检索层
- 先完善 current_state schema

### Gate 2：历史召回是否明显不足？

如果不足：
- 先判断是事件对象设计不合理
- 还是纯向量检索不够
- 再决定是否升级为混合检索或 MemPalace 风格

### Gate 3：Token 是否仍然过高？

如果仍然过高：
- 先收紧最近对话窗口和检索预算
- 不要先盲目增加总结层数

***

## 十二、一句话总结

本方案的核心不是“再造一个更大的记忆桶”，而是：

> 用常驻稳定层固定角色与长期边界，  
> 用当前状态层固定此刻局面，  
> 用历史检索层按需回忆过去，  
> 用最近对话窗口保证局部连贯性。

只有把这四种职责真正拆开，FoxChat 的记忆系统才有可能既降低 Token，又提升长期一致性与可控性。
