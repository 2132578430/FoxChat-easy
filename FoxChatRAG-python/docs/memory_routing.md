# 信息路由规则文档

> 定义 FoxChat 记忆架构中信息分流判断顺序、决策表与调试要求
> 版本：第一版
> 更新时间：2026-04-27

---

## 一、分流判断顺序

**核心原则：先判 A2，再判 B，最后判 C**

```text
收到新信息候选
   ↓
1. 判断是否命中高优先级边界规则 → A2
   ↓ (未命中)
2. 判断是否属于当前有效状态 → B
   ↓ (未命中)
3. 判断是否具备事件意义 → C
   ↓ (未命中)
丢弃或忽略
```

**判断顺序理由：**
- 只要命中高优先级边界规则，优先进入 A2，不应只当作当前情绪或普通事件
- 未命中 A2 时，再判断是否属于当前有效状态
- 既不是 A2 也不是当前状态的内容，再进入历史事件层

## 二、A2/B/C 分流决策表

| 输入特征 | 优先去向 | 可否同时写入其他层 | 说明 |
|----------|----------|--------------------|------|
| 用户明确说"不要这样叫我/不要聊这个/不要这样回应我" | A2 | 可同步生成B层短期状态，但A2为主 | 高优先级边界 |
| 用户表达稳定偏好，如"我更喜欢你直接一点" | A2候选 | 可同步生成C事件；不默认直接写B | 证据不足时先保留为候选 |
| 用户当前情绪明显变化，如"我现在很难过/放松多了" | B | 可同步生成C事件（若具备事件意义） | 情绪变化首先服务当前回复 |
| 用户提到一段过去经历，但没有表达长期偏好或当前状态变化 | C | 不默认写A2/B | 纯历史信息优先留在事件层 |
| 用户说"上次那件事后来解决了" | C | 若同时完成了承诺事项，可更新B的`unfinished_items` | 事件续写 |
| 用户说"以后别再提这个了，我每次都会难受" | A2 | 可同步写C；必要时同步写B的情绪状态 | 长期禁忌优先 |
| 用户说"这两天我都在焦虑考试" | B | 可同步写C | 当前焦点+情绪为主 |
| 用户说"我一直都不喜欢别人催我" | A2候选 | 可同步写C | 带有长期性表达，但需验证稳定性 |
| AI做出明确承诺，如"明天我再陪你看结果" | B | 可同步写C | 先写`unfinished_items` |
| 一次性抱怨或上下文拒绝，如"今天不想聊这个" | B或忽略 | 不进入A2，必要时可写C | 单轮有效不应误升为长期边界 |

## 三、多层并存规则

### 3.1 允许的并存方式

- **A2 + C**：可并存，如"不要叫我宝宝"可主写A2，辅写C作为来源事件
- **B + C**：可并存，如当前情绪+历史事件背景
- **A2 + B**：可并存但职责不混淆，A2是长期约束，B是当前局面

### 3.2 禁止的情况

- **三层全写满**：不应同时在A2、B、C都注入完整版本，避免重复污染Prompt
- **职责混淆**：A2与B的职责不能互相替代

### 3.3 各层职责定位

| 层级 | 职责定位 |
|------|----------|
| A2 | "以后持续有效的约束或画像" |
| B | "当前几轮内影响回复方式的局面" |
| C | "过去发生过、未来按需回忆的信息" |

### 3.4 实现顺序建议

```text
1. 先做 rule-based 分流
2. 再做候选去重
3. 最后做各层写回
```

不建议先统一落库后再二次猜测归属，否则容易出现多层重复。

## 四、A2 提升条件

### 4.1 立即进入 A2（无需重复出现）

**命中条件：用户明确表达高优先级边界**

| 边界类型 | 示例 |
|----------|------|
| 称呼禁忌 | "不要叫我宝宝" |
| 话题禁忌 | "不要聊这个话题" |
| 互动方式禁忌 | "不要这样回应我" |
| 隐私边界 | 明确的隐私边界表达 |
| 身体/亲密边界 | 明确的身体/亲密边界表达 |

**判断原则：**
- 以用户直接表达为主
- 不要求多轮重复确认
- 命中固定规则类目后立即进入A2

### 4.2 延迟进入 A2（需要稳定性证据）

**长期偏好、稳定互动习惯、长期画像特征**

| 提升条件 | 说明 |
|----------|------|
| 连续2个summary batch提取同一结论 | 稳定性证据 |
| 最近30轮内>=3次一致表达 | 且无明显冲突证据 |

### 4.3 不进入 A2 的情况

| 情况 | 说明 |
|------|------|
| 一次性情绪发泄 | 如临时抱怨 |
| 上下文性短期抱怨 | 仅针对当前情境 |
| 仅对当前轮有效的拒绝或偏好 | 如"今天不想聊这个" |
| 随任务变化的临时需求 | 不具长期性 |

## 五、高优先级边界判定规则

### 5.1 固定规则类目

| 类目 | 判断依据 |
|------|----------|
| 称呼禁忌 | 用户明确说"不要叫我XX" |
| 创伤/触发话题 | 用户明确说"不要聊XX话题" |
| 隐私边界 | 用户明确表达隐私边界 |
| 身体/亲密边界 | 用户明确表达身体/亲密边界 |
| 明确拒绝的互动方式 | 用户明确说"不要这样回应我" |

### 5.2 判断原则

```text
user_explicit 来源 > summary_inference 来源
"长期稳定" = 重复证据规则，不是抽象语义判断
"高风险踩雷" = 命中固定规则类目，不是主观评估
```

### 5.3 冲突处理

| 冲突情况 | 处理方式 |
|----------|----------|
| 新候选与已有A2冲突 | 不直接覆盖，先标记冲突候选，等待下一次batch确认 |
| 稳定偏好类A2后续有反向证据 | 允许因多次反向证据降级回普通profile |
| 高优先级边界类A2 | 默认不自动降级 |

## 六、调试输出要求

### 6.1 必须记录的字段

每个候选分流时至少记录：

| 字段 | 说明 |
|------|------|
| `candidate_type` | 候选类型（如：boundary、emotion、event） |
| `target_layer` | 目标层级（A2、B、C） |
| `why_routed` | 分流理由（简短说明） |
| `source_round` | 来源轮次 |

### 6.2 调试输出示例

```json
{
  "candidate_type": "boundary",
  "target_layer": "A2",
  "why_routed": "命中称呼禁忌规则，用户明确表达",
  "source_round": 42
}
```

### 6.3 调试目的

阶段3调试时应能看到"为什么进入A2/B/C"，否则后续很难解释误判。

## 七、C层兜底原则

**C层是兜底层，不是垃圾桶**

| 原则 | 说明 |
|------|------|
| 不能不确定就丢到C | 需先判断是否命中A2/B |
| C只保留有事件意义的内容 | 未来可能需要按需回忆 |
| 不承担当前状态职责 | 当前状态应优先进入B |
