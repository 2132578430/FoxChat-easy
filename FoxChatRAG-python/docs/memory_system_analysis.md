## 一、项目概述

### 1.1 项目架构

FoxChat 是一个基于 Spring Boot + Python 的智能聊天应用，采用微服务架构：

```
┌─────────────────────────────────────────────────────────────────┐
│                        FoxChat 系统架构                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Java 端 (Spring Boot 3.4.3):                                  │
│  • foxChat-web: REST API 控制器                                │
│  • foxChat-service: 业务逻辑、数据访问                          │
│  • foxChat-pojo: 数据模型                                       │
│  • foxChat-netty: WebSocket 通讯                               │
│  • foxChat-common: 公共工具                                     │
│                                                                 │
│  Python 端 (FastAPI + LangChain):                              │
│  • LLM 调用、记忆管理、RAG 检索                                 │
│  • 多模型支持：DeepSeek, Kimi, Claude, GLM, MiniMax            │
│                                                                 │
│  数据存储:                                                      │
│  • MySQL: 持久化数据                                            │
│  • Redis: 缓存、对话状态、记忆存储                              │
│  • Chroma: 向量数据库（对话总结）                               │
│  • Minio: 文件存储                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 二、当前记忆系统实现

### 2.1 多层记忆架构

项目实现了五层记忆系统：

| 记忆层         | 存储位置       | 存储内容                    | 加载策略         |
| ----------- | ---------- | ----------------------- | ------------ |
| 原始经历        | Redis      | 用户创建角色时的原始描述            | 初始化时         |
| 角色核心锚点      | Redis      | 100-200字核心描述 + 绝对边界     | 每次对话必加载      |
| 用户画像        | Redis      | 7维度结构化画像（JSON）          | 每次对话必加载      |
| 角色卡         | Redis      | SillyTavern 格式角色卡（JSON） | 每次对话必加载      |
| Memory Bank | Redis      | 事件和状态（JSON数组，最多50条）     | 每次对话必加载      |
| Chroma 向量库  | Chroma     | 对话总结（每6轮总结一次）           | 每次对话必加载（k=5） |
| 最近对话        | Redis List | 最近30条对话                 | 每次对话必加载      |

### 2.2 记忆更新流程

每6轮对话后触发记忆更新：

```
对话触发总结
    │
    ▼
取出最近消息（保留最近10条）
    │
    ▼
总结消息存入 Chroma 向量库
    │
    ▼
提取关键事件存入 Memory Bank
    │
    ▼
检查并压缩 Memory Bank（超过50条压缩到30条）
    │
    ▼
更新用户画像
```

## 三、核心问题分析

### 3.1 问题一：自我审查悖论

**问题描述：**

- 模型既要生成回复，又要自己检查自己
- 导致上下文错乱、逻辑错误
- 容易被用户带偏

**举例：**

```
用户说："是用户要去洗漱了而不是模型要去洗漱"
模型默认："好的，你说得对" ← 缺乏独立判断能力
```

### 3.2 问题二：记忆压缩导致信息丢失

**问题描述：**

- 每6轮对话总结一次，总结后丢失细节
- Memory Bank 压缩时丢失主体归属信息
- 多次对话后记忆饱和，Token 消耗急速上升

**举例：**

```
原始对话："我要去洗漱了" (用户)
    ↓ 总结压缩
压缩结果："很开心，可以洗漱了" ← 丢失了【谁】要去洗漱
    ↓ 后续对话
用户纠正："是用户要去洗漱"
    ↓ 模型反应
模型默认："好的，你说得对" ← 因为记忆里没有主体归属信息
```

### 3.3 问题三：Token 消耗不可控

**实际测量数据（使用 tiktoken 精确计算）：**

| 部分                                     | Token 数量     |
| -------------------------------------- | ------------ |
| character\_card                        | 812          |
| core\_anchor                           | 398          |
| role\_emotion\_state                   | 160          |
| user\_profile                          | 451          |
| **固定部分小计**                             | **1821**     |
| 其他固定部分 (soul, role\_declaration, etc.) | \~500        |
| **固定部分总计**                             | **\~2321**   |
| recent\_msg (12条)                      | 3644 ← 问题所在！ |
| memory\_bank (9条)                      | 360          |
| Chroma 检索 (k=5)                        | \~250        |
| **动态部分总计**                             | **\~4004**   |
| **每次对话总Token**                         | **\~6325**   |

**关键发现：**

- recent\_msg 只有 12 条，但 Token 消耗高达 3644
- 平均每条对话 \~303 Token（内容很长）
- 动态部分占比 63%
- 如果不限制，100条消息 ≈ 30000 Token

### 3.4 问题四：Chroma 向量库效果不佳

**问题描述：**

- Chroma 存储的是对话总结，和 Memory Bank 内容重叠
- 检索效果不好，用户反馈"跟摆设没区别"
- 每次对话都检索，没有懒加载机制

## 五、可能的改进方向

### 5.1 方向一：减少动态部分 Token 消耗

**方案：**

- 减少 recent\_msg 条数（12条 → 6条）
- 减少 memory\_bank 条数（50条 → 15条）
- Chroma 懒加载（只有用户提到过去时才检索）

**预期效果：**

- Token 从 \~6325 减少到 \~4621
- 节省约 30%

**问题：**

- 减少条数可能导致信息丢失
- 懒加载可能导致召回率下降

### 5.2 方向二：借鉴 SillyTavern 的永久记忆方案

**SillyTavern 的核心思路：**

- 不限制记忆数量，通过向量检索按需获取
- 每条消息都存储到向量数据库
- 每次对话只检索最相关的 k 条记忆
- Token 消耗稳定，不随记忆增长

**实现方式（参考）：**

- Arkhon Memory: FAISS 向量数据库，每条消息自动嵌入
- Qdrant Memory: 每个角色独立 collection，实时检索
- Memory Books: 结构化总结存储到 Lorebook

**问题：**

- 向量检索的召回率如何保证？
- 用户说"上次那个事"，如何精准检索到相关记忆？

### 5.3 方向三：结构化记忆存储

**方案：**

- 总结时输出结构化 JSON（包含 actor, action, keywords）
- 整体存储到 Chroma（不拆分）
- 混合检索：向量检索 + BM25 + 时间窗口 + Rerank

**结构化格式示例：**

```json
{
  "summary_id": "sum_001",
  "timestamp": "2024-01-15T22:30:00",
  "topic": "睡前准备",
  "events": [
    {"actor": "USER", "action": "go_to_wash", "content": "用户要去洗漱了"},
    {"actor": "AI", "action": "wait", "content": "AI表示等待"}
  ],
  "keywords": ["洗漱", "睡前", "等待"]
}
```

**问题：**

- 实现复杂，效果难以预测
- 混合检索需要额外开发

### 5.4 方向四：引入成熟框架

**可选框架：**

- MemGPT / Letta: 专门的记忆管理 Agent
- LangGraph: 有状态的 Agent 工作流

**问题：**

- 需要引入新框架，与现有架构整合有成本
- 增加系统复杂度

## 六、核心问题总结

### 6.1 根本矛盾

```
保留更多信息 ←─────────────────────→ 减少Token消耗
      │                                    │
      ▼                                    ▼
记忆更准确                            成本更低
但Token爆炸                           但记忆丢失
```

### 6.2 关键问题

1. **记忆压缩导致信息丢失**：主体归属、时间细节等关键信息在压缩后丢失
2. **Token 消耗不可控**：recent\_msg 是最大消耗源，每条对话 \~303 Token
3. **向量检索召回率**：不全量加载时，如何保证相关记忆能被检索到
4. **自我审查悖论**：单一模型无法客观评估自己

3. 结构化记忆存储是否是正确的方向？
4. 四角色架构是否能解决自我审查问题？记忆问题是否要先解决？

## 八、附录：SillyTavern 记忆实现参考

### 8.1 Arkhon Memory

- 每条消息自动嵌入（转换为向量）
- 存储在本地 FAISS 向量数据库
- 根据当前对话语义检索相关记忆
- 100% 本地，不上传云端
- 记忆永久保存，不会丢失

### 8.2 Qdrant Memory

- 使用 Qdrant 向量数据库
- 每个角色有独立的 collection（完全隔离）
- 自动保存对话，实时嵌入
- 生成前检索相关记忆，注入 prompt

### 8.3 Memory Books

- 将记忆存储为 Lorebook entries
- 支持场景标记（手动选择对话范围）
- AI 生成 JSON 格式的总结
- 支持多层记忆合并（Scene → Summary）

## 九、SillyTavern 核心机制详解

### 9.1 关键词触发机制（World Info / Lorebook）

**核心原理：关键词是用户手动定义的，不是自动提取的！**

SillyTavern 的 Lorebook 条目结构：

```
条目 #1
─────────────────────────────────────────
Key (关键字): "日本, Japan, 东京"     ← 用户手动定义
Content: "用户和AI上次去了日本，去了东京塔..."
Position: After Character Definition
Order: 100
```

**关键词来源：**

- 用户手动输入（不是自动提取）
- 可以是普通文本："日本", "洗漱"
- 可以是正则表达式：`/(?:去|来)日本/`
- 多个关键词用逗号分隔

**扫描流程：**

```
用户输入："上次去日本的时候..."

Step 1: 扫描最近 N 条消息（Scan Depth）
─────────────────────────────────────
• Scan Depth = 5（扫描最近5条消息）
• 检查这些消息中是否包含任何条目的关键词

Step 2: 关键词匹配
─────────────────────
• 消息中包含"日本"
• 条目 #1 的 Key 包含"日本"
• 匹配成功 → 条目 #1 被激活

Step 3: 注入 Content
─────────────────────
• 条目 #1 的 Content 被注入到 prompt
• "用户和AI上次去了日本，去了东京塔..."
```

**为什么关键词可以获取相关记忆？**

- 因为关键词是用户精心设计的
- 用户知道哪些词会触发这条记忆
- 类似"标签"系统

### 9.2 递归扫描机制

**核心原理：检索到记忆后，扫描内容中的关键词，再用这些关键词去检索其他条目**

**递归扫描流程：**

```
条目 #1:
  Key: "Bessie"
  Content: "Bessie is a cow and is friends with Rufus."

条目 #2:
  Key: "Rufus"
  Content: "Rufus is a dog."

用户输入："Bessie is hungry"

Round 1: 首次扫描
─────────────────────────────────────────
• 扫描用户输入，发现关键词 "Bessie"
• 条目 #1 被激活
• 注入 Content: "Bessie is a cow and is friends with Rufus."

Round 2: 递归扫描（扫描已激活条目的 Content）
─────────────────────────────────────────────
• 扫描条目 #1 的 Content
• 发现 Content 中包含 "Rufus"
• 条目 #2 的 Key 包含 "Rufus"
• 条目 #2 被激活（递归）
• 注入 Content: "Rufus is a dog."

Round 3: 继续递归
─────────────────────────────────────────
• 扫描条目 #2 的 Content
• 没有发现新的关键词
• 递归结束

最终注入：
• 条目 #1: "Bessie is a cow and is friends with Rufus."
• 条目 #2: "Rufus is a dog."
```

**递归控制：**

- 递归深度限制：默认 3 次，最多 10 次
- 防止无限递归
- Token 预算在每次递归时都会检查

**每个条目的递归控制选项：**

| 选项                    | 说明                              |
| --------------------- | ------------------------------- |
| Non-recursable        | 禁止被其他条目递归激活                    |
| Delay until recursion | 只在递归时激活，不在首次扫描时激活              |
| Prevent further recursion | 激活后不再触发其他条目                   |

### 9.3 多种机制组合

SillyTavern 不是单纯依赖向量检索，而是多种机制组合：

| 机制            | 说明                              | 作用         |
| ------------- | ------------------------------- | ---------- |
| 关键词触发         | 用户手动定义关键词                       | 精准召回       |
| 递归扫描          | 扫描已激活条目的 Content，触发其他条目        | 自动关联相关记忆   |
| BM25 模糊搜索     | 补充精确关键词匹配                       | 解决关键词遗漏    |
| 时间衰减          | 旧记忆权重降低，新记忆权重提高                | 更自然的记忆行为   |
| 冷却/预热         | 防止重复注入、防止偶然触发                  | Token 消耗控制 |
| 条目衰减和新鲜度      | 旧条目获得 boost，频繁触发的条目获得 penalty  | 自然轮换记忆     |
| Token 预算控制    | 只有触发的条目才注入，预算耗尽后停止             | Token 消耗可控 |

### 9.4 Chat Vectorization（消息分块 + Shuffling）

**核心原理：将相关历史消息移动到对话历史的开头或结尾**

```
为什么移动到开头/结尾？
─────────────────────────────────────────
• 开头和结尾的消息对模型回复影响最大
• 中间的消息影响较小
• 将相关消息移动到这些位置，让模型更关注
```

**实现方式：**

- 每条消息分块存储（chunk size = 400 字符）
- 使用最近 2 条消息作为查询
- 检索相关历史消息
- 将相关消息"移动"到对话历史的开头或结尾

### 9.5 SillyTavern vs 自动提取关键词

| 方面            | SillyTavern                     | FoxChat 可借鉴的方式          |
| ------------- | ------------------------------- | ------------------------ |
| 关键词来源         | 用户手动定义                          | LLM 自动提取                 |
| 维护成本          | 需要人工维护                          | 自动适应对话内容                 |
| 精准度           | 高（用户精心设计）                       | 取决于 LLM 提取质量            |
| 递归机制          | 扫描 Content 中的关键词               | 可实现类似机制                  |

**FoxChat 可借鉴的实现方式：**

```
对话总结 → LLM 自动提取关键词 → 存储 keywords 字段
    ↓
检索时：关键词匹配 + 向量检索 + Rerank
    ↓
递归：扫描已激活记忆的 Content，寻找其他关键词
```

## 十、关键洞察

### 10.1 SillyTavern 为什么召回率高？

1. **关键词是用户精心设计的**：用户知道哪些词会触发这条记忆
2. **递归可以自动关联相关记忆**：不需要用户提到所有关键词
3. **多种机制组合**：关键词 + BM25 + 向量检索 + 时间衰减
4. **Token 预算控制**：只有触发的条目才注入

### 10.2 FoxChat 可以借鉴什么？

1. **结构化记忆存储**：包含 keywords 字段
2. **关键词触发机制**：先关键词匹配，再向量检索
3. **递归扫描**：扫描已激活记忆的 Content
4. **Token 预算控制**：只有触发的记忆才注入

### 10.3 需要解决的技术问题

1. **关键词自动提取**：如何用 LLM 提取高质量关键词？
2. **关键词匹配效率**：如何高效匹配大量关键词？
3. **递归深度控制**：如何防止无限递归？
4. **Token 预算分配**：如何合理分配 Token 预算？
