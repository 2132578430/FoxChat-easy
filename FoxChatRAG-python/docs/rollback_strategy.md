# 个人项目回退策略

> 适用项目: FoxChatRAG-python
> 前提: 个人项目，非生产环境，已有 git 备份

---

## 设计决策

本项目采用"备份 + 明确回退步骤"替代生产级双架构切换开关。

**原因:**
- 项目为个人项目，无需灰度发布或长期并行回滚
- 已有 git 备份和阶段性提交，可直接回退到稳定版本
- 不增加运行时切换逻辑，降低代码复杂度

---

## 回退操作步骤

### 小范围回退（单次修改）

1. **发现问题后立即停止新修改**
   - 不要继续提交新的变更
   - 记录问题现象和触发条件

2. **回退到上一个稳定提交**
   ```bash
   # 查看最近的提交历史
   git log --oneline -10
   
   # 回退到指定提交（保留工作区修改）
   git reset --soft <stable-commit-hash>
   
   # 或完全回退（丢弃工作区修改）
   git reset --hard <stable-commit-hash>
   ```

3. **验证回退后的行为**
   - 运行基本对话测试
   - 检查 Prompt 输出是否恢复预期结构

4. **重新开始修复**
   - 在新分支上尝试修复方案
   - 验证后再合并到主分支

### 大范围回退（整个阶段）

1. **找到阶段开始前的提交**
   ```bash
   # 查找特定阶段的提交边界
   git log --oneline --grep="阶段5" --grep="阶段6" --grep="阶段7"
   
   # 或使用标签
   git tag -l
   ```

2. **创建回退分支**
   ```bash
   # 从稳定点创建新分支
   git checkout -b rollback-memory-v1 <stable-commit-hash>
   ```

3. **在回退分支上验证**
   - 确保核心功能正常
   - 确认问题是否消失

4. **决定后续策略**
   - 若问题严重，可在回退分支上重新开始
   - 若问题可控，可在主分支上局部修复

## 备份策略

### 提交前检查

每次重大修改前，执行以下检查：

1. **当前状态确认**
   ```bash
   git status
   git diff --stat
   ```

2. **功能基本验证**
   - 普通对话能正常响应
   - Prompt 结构符合预期

3. **提交记录**
   ```bash
   git add -A
   git commit -m "feat: <描述修改内容>"
   ```

### 阶段性备份

每完成一个阶段（如阶段 5、阶段 6），创建标签：

```bash
git tag -a "memory-v1-phase5-done" -m "阶段5完成时的稳定版本"
git tag -a "memory-v1-phase6-done" -m "阶段6完成时的稳定版本"
```

## 问题归因流程

### 发现问题后的记录格式

```markdown
## 问题记录

- **发现时间**: YYYY-MM-DD HH:MM
- **触发场景**: <描述输入和预期>
- **实际行为**: <描述异常输出>
- **问题层**: A2 / B / T / C / D / Prompt Layout
- **严重程度**: 高 / 中 / 低
- **可能原因**: <初步判断>
- **回退建议**: 是否需要回退，回退到哪个提交
```

### 问题追踪文件

所有问题记录写入 `docs/memory_validation_issues.md`，按优先级排序修复。

---

## 验收通过后的归档

当记忆架构 V1 最终通过验收：

1. 创建最终标签
   ```bash
   git tag -a "memory-v1-complete" -m "记忆架构V1正式完成"
   ```

2. 更新文档
   - `docs/memory_architecture_completion_status.md` 标记为完成
   - `docs/memory_validation_issues.md` 关闭所有问题

3. 清理临时分支
   ```bash
   git branch -d rollback-memory-v1  # 若已合并或不再需要
   ```

---

## 不做的事

- 不实现运行时架构切换开关
- 不实现双写或并行比较机制
- 不保留多版本 Prompt 在同一代码中切换
- 不引入灰度发布逻辑

---

## 责任声明

本回退策略仅适用于个人项目场景。若项目转为生产环境或多人协作，需重新评估切换机制需求。
