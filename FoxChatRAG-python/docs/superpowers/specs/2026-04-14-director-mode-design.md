## 1. 架构概览

### 1.1 角色定义

| 角色 | 核心任务 | 模型选择 | 调用次数 |
|------|---------|---------|---------|
| **导演 (Director)** | 信息蒸馏：从全量上下文提取精简指令 | 大模型(minimax M2.7) | 1次 |
| **演员 (Actor)** | 内容执行：按导演指令生成台词+动作 | 大模型(minimax M2.7) | 1-2次 |
| **审查员 (Reviewer)** | 质量兜底：校验OOC、语法错误 | 大模型(minimax M2.7) | 1-2次 |

### 1.2 信息流

```
用户输入
    ↓
┌─────────────────────────────────────────────────────────┐
│                     【导演】Director                       │
│  输入：角色卡 + 用户画像 + 记忆 + 历史对话                  │
│  输出：场景判断 + 情绪基调 + 回复铁则                       │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                     【演员】Actor                          │
│  输入：极简人设 + 导演输出 + 最近对话                       │
│  输出：带<action>标签的台词                                │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                     【审查员】Reviewer                     │
│  输入：导演铁则 + 待审查内容 + 用户输入                     │
│  输出：通过 / 打回 / 降级修正                              │
└─────────────────────────────────────────────────────────┘
                            ↓
                      MessageBlock
```

## 2. 各角色职责

### 2.1 导演 (Director)

**必须注入：**
- 角色核心人设（3-5条铁则）
- 用户画像（2-3条）
- 强相关记忆（≤3条）：Chroma检索长期记忆 + Redis memory_bank全部注入
- 最近3-5轮对话
- 用户当前输入

**绝对不注入：**
- 长篇角色卡
- 全量记忆库
- 过期对话
- 示例对话

**输出格式：**
```json
{
  "scene": "场景判断（如：撒娇、安慰、调侃）",
  "emotion_base": "情绪基调（如：开心、温柔、略带无奈）",
  "rules": ["回复铁则1", "回复铁则2", "回复铁则3"]
}
```

### 2.2 演员 (Actor)

**必须注入：**
- 极简人设（1句话）
- 导演完整输出（场景 + 情绪 + 铁则）
- 最近3轮对话
- 用户输入

**绝对不注入：**
- 全量人设
- 全量记忆
- 多余历史

**输出格式：**
- 带`<action>`标签的台词
- 动作嵌入在标签中，如：
  ```
  <action>微笑</action>你真棒呀，宝贝~
  <action>叹气</action>好吧，那下次再说吧
  ```

### 2.3 审查员 (Reviewer)

**必须注入：**
- 导演铁则（rules数组）
- 待审查内容（演员输出）
- 用户输入

**绝对不注入：**
- 任何多余背景
- 记忆
- 历史

**输出判断：**
- 通过：内容符合要求
- 打回：内容有OOC或严重语法问题，需演员重生成
- 降级：二次审查仍有问题，审查员只修语法错误，逻辑/场景错误不改

## 3. 处理流程

```
用户输入
    ↓
【导演】处理 → 输出：{scene, emotion_base, rules}
    ↓
【演员】生成 → 输出：带<action>标签的台词
    ↓
【审查员】首次审查
    ↓
通过？ → 是 → 输出
    ↓ 否
打回次数 < 1？ → 是 → 【演员】重生成 → 【审查员】再审
    ↓
    ↓ 否（已打回1次）
降级处理 → 【审查员】只修语法错误 → 输出
                ↓
            逻辑/场景错误 → 不改 → 直接输出
```

**调用次数：**
- 正常：3次（导演 + 演员 + 审查员）
- 最差：4次（导演 + 演员 + 审查员 + 演员重生成 + 审查员再审）

## 4. 记忆系统

### 4.1 导演记忆注入

导演从两个来源获取记忆：

1. **Chroma 向量检索**
   - 每6轮（约12条消息）总结一次存入
   - 按用户输入语义检索强相关记忆
   - 取≤3条

2. **Redis memory_bank**
   - 事件和状态已精选
   - 当前数据量3-4条，可全部注入
   - 有time标注（早期/最近），导演可判断时效性

### 4.2 结论

当前6轮总结频率够用，无需调整。记忆系统设计保持现状，先观察实际效果。

## 5. 错误处理

| 阶段 | 错误类型 | 处理方式 |
|------|---------|---------|
| 导演 | 无法理解上下文 | 使用默认场景+中性情绪 |
| 演员 | 生成内容为空 | 返回预设回复（如："嗯..."） |
| 审查员 | LLM调用失败 | 跳过审查，直接输出演员内容 |
| 打回重生成 | 演员返回空/无效 | 审查员降级处理，只修复语法 |

## 7. 关键设计决策

1. **动作师合并到演员**：演员同时输出台词+动作，审查员统一审查
2. **审查员可打回1次**：打回后审查员再审，第二次仍有问题则降级处理
3. **降级策略**：审查员只修语法错误，逻辑/场景错误不改，直接输出
4. **模型统一**：全部使用minimax M2.7，通过global-settings配置方便切换
5. **输出格式保持**：与普通聊天共用MessageBlock格式

## 8. 与情绪系统的关系

- 情绪系统（SillyTavern风格）是独立模块，与四角色架构无关
- 两个系统独立并行，最终输出合并到同一个MessageBlock
- 如需合并，可在后续按需调整
