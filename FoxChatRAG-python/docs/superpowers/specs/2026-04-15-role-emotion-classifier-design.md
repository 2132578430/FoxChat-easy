## 1. 背景与目标

### 1.1 背景

当前项目（FoxChatRAG）有多层记忆系统（Soul、Core Anchor、Character Card、Memory Bank、User Profile），但缺少**动态情绪状态**机制。

用户希望通过学习情绪分类功能，掌握：

- LLM 的情绪分类能力
- 事件驱动的状态管理系统
- 后续可扩展的情感分类模型集成

### 1.2 核心目标

**学习目的**：验证 LLM 情绪分类在中文对话中的效果。

**非目标**（本期不实现）：

- 情绪状态不影响角色回复
- 复杂的衰减机制
- 专门的情绪分类模型

***

## 2. 设计方案

### 2.1 方案 B：基于模型输出 + 异步执行

**核心逻辑**：分析模型的回复内容，判断角色当前的情绪状态。

**原因**：
- 情绪是角色的内在状态，不是用户输入的函数
- 模型回复时已经"扮演"了角色，情绪自然流露
- 符合"拟人化"目标（角色有独立的情绪连续性）

### 2.2 执行流程

```
用户输入
    ↓
读取 Redis 当前情绪状态（上一轮更新的）
    ↓
模型生成回复
    ↓
立即返回回复给用户 ← 用户无感知
    ↓（后台异步）
情绪分类服务（分析模型回复）
    ↓
情绪状态更新逻辑
    ↓
记录到 Redis / 日志（供下一轮使用）
```

### 2.3 与消息总结的并行关系

**每轮对话的后台任务：**

| 任务 | 触发频率 | 执行方式 | 目的 |
|------|----------|----------|------|
| 消息总结 | 每 6 轮 | 异步 | 更新 Memory Bank |
| 情绪判断 | 每轮 | 异步 | 更新情绪状态 |

**并行执行，互不影响：**

```python
async def background_tasks():
    try:
        await classify_emotion(model_reply)
    except Exception as e:
        logger.error(f"情绪判断失败: {e}")
    
    try:
        await summary_msg(...)
    except Exception as e:
        logger.error(f"消息总结失败: {e}")
```

### 2.4 情绪状态的时间关系

| 轮次 | 用户输入 | 模型回复 | 情绪判断（异步） | Redis 状态 |
|------|----------|----------|------------------|------------|
| 第1轮 | "你好" | "你好呀！" | 开心:确定 | 初始 → 开心 |
| 第2轮 | "今天天气真好" | "是啊，阳光明媚！" | 开心:确定 | 开心 → 开心 |
| 第3轮 | "我失恋了" | "啊...我很抱歉..." | 悲伤:确定 | 开心 → 悲伤 |
| 第4轮 | "算了，不想了" | "嗯，向前看吧" | neutral:确定 | 悲伤 → neutral |

**关键理解：**
- 第 N 轮的情绪判断，更新的是第 N+1 轮可用的状态
- 第 N 轮的模型回复，使用的是第 N-1 轮更新后的情绪状态

## 3. 情绪分类服务

### 3.1 职责

- 封装情绪分类调用
- 支持多模型切换（通过配置）
- 返回情绪标签 + 确定性

### 3.2 接口设计

```python
async def classify_emotion(model_reply: str) -> tuple[str, str]:
    """
    分析模型回复的情绪状态
    
    Args:
        model_reply: 模型的回复文本
    
    Returns:
        (emotion_label, certainty): 情绪标签和确定性
    """
    ...
```

### 3.3 情绪状态更新规则

```
情绪分类器返回结果
    ↓
certainty == "确定" ？
    ↓ yes       ↓ no
更新情绪状态    保持原情绪
```

**原因**：避免模型回复中性内容时，角色情绪被强制变为 neutral。

## 4. 情绪状态存储

### 4.1 Redis Key

```python
# Key: chat:memory:{user_id}:{llm_id}:role_emotion_state
{
    "emotion": "开心",
    "certainty": "确定",
    "last_update": "2026-04-15T10:30:00",
    "last_trigger": "太好了！我也为你感到高兴！"
}
```

### 4.2 字段说明

| 字段 | 说明 |
|------|------|
| `emotion` | 当前情绪标签 |
| `certainty` | 确定性（确定/不确定） |
| `last_update` | 最后更新时间 |
| `last_trigger` | 触发情绪变化的模型回复片段 |

## 5. 模型配置

### 5.1 全局配置

```python
# settings.py

```

### 5.2 支持的模型

| 配置值 | 模型 | 说明 |
|--------|------|------|
| `minimax` | MiniMax API | 默认，中文支持好 |
| `qwen3b` | Qwen2.5-3B-Instruct | 本地，中文优化 |
| `qwen7b` | Qwen2.5-7B-Instruct INT4 | 本地，能力强 |

### 5.3 模型切换

通过修改 `.env` 文件即可切换：

```bash
# .env
EMOTION_MODEL=minimax
```

## 7. 日志与验证

### 7.1 日志格式

```
【情绪分类】模型回复: "太好了！我也为你感到高兴！" → 开心:确定 [确定性足够，更新状态]
【情绪分类】模型回复: "嗯，好的。" → neutral:不确定 [确定性不足，保持状态: 开心]
```

### 7.2 Redis 记录

```json
{
    "timestamp": "2026-04-15T10:30:00",
    "model_reply": "太好了！我也为你感到高兴！",
    "emotion_label": "开心",
    "certainty": "确定",
    "action": "updated"
}
```

### 7.3 验证方式

- 通过日志观察情绪分类是否合理
- 通过 Redis 检查情绪状态变化是否符合预期
- 验证情绪连续性（角色情绪是否自然过渡）

***

## 8. 错误处理

| 异常情况 | 处理方式 |
|----------|----------|
| 模型调用失败 | 记录日志，保持原情绪状态 |
| 分类超时 | 跳过本次分类，不影响对话 |
| Redis 写入失败 | 跳过，不影响对话 |
| 解析结果失败 | 保持原情绪状态 |

***

## 9. 情绪分类 Prompt（minimax）

```markdown
分析角色回复中流露的情绪状态，只输出一个词和确定性程度。

情绪选项：开心、悲伤、愤怒、惊讶、恐惧、厌恶、neutral

确定性选项：确定、不确定

规则：
- 如果角色回复有明显情绪表达，选择对应情绪 + 确定性
- 如果角色回复是中性表达，选择 neutral + 确定性
- 只输出"情绪:确定性"，不要其他内容

角色回复：{model_reply}

输出：
```

**输出示例**：

- `开心:确定` → 角色回复有明显开心情绪，更新状态
- `开心:不确定` → 角色回复情绪不明显，保持原状态
- `neutral:确定` → 角色回复是中性表达，更新为 neutral
